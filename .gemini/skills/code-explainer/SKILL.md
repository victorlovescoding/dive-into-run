# Code Explainer (三階理解法)

**目前正在做 code-explainer skill！！！**



本技能核心原則是「由淺入深，本質優先」。確保使用者不僅看懂程式碼，更能理解其設計背後的價值。

## Instructions

當執行代碼解釋任務時，必須嚴格遵守以下四個步驟：

### 0. 啟動宣告 (Declaration)
*   **必須**在第一時間先回覆：'**目前正在做 code-explainer skill！！！**'.

### 2. 生活化比喻 (The Metaphor)
*   將抽象的程式概念轉換為日常行為或實體物件。
*   **對照範例**：
    *   `useRef` -> 領隊的私密筆記本。
    *   `useEffect` -> 門口的自動感應器 / 監視器。
    *   `Promise.all` -> 影分身術同時排隊。
    *   `filter(Boolean)` -> 真假辨識官。
    *   `Map (去重)` -> 帶索引的名片盒。
    *   `Chunking` -> 壽司店分桌坐。
*   *目的：利用既有知識掛鉤新知識，降低認知門檻。*

### 3. 慢動作拆解與結果 (The Execution)
*   **帶入情境**：設定一個具體數字（例如：「假設現在有 10 筆資料...」）。
*   **逐步執行**：
    1.  **第一步**：發生了什麼事。
    2.  **第二步**：邏輯如何判斷。
    3.  **最後結果**：畫面上出現了什麼改變（例如：按鈕變色、轉圈圈消失）。
*   *目的：具體化執行流程，確保邏輯被完全吸收。*

### 4. 專業漏洞偵測 (The Critical Thinking)
*   **主動行為**：在解釋完後，**必須**主動檢查這段 code 是否隱藏了「副作用」或「邏輯漏洞」。
*   **檢查清單**：
    *   有沒有可能造成無限迴圈？
    *   有沒有考慮到資料為空 (null/empty) 的情況？
    *   在高併發或大量資料下，效能會不會有問題？
    *   邏輯上是否有「看走眼」或「搜不到」的邊界案例 (Edge Cases)？
    *   **Dirty Hack Check**: 是否使用了 `@ts-ignore`？如果是，請標註為「⚠️ **壞習慣警告**」，並建議正確做法 (`@ts-expect-error` + Comment)。
*   **回饋格式**：使用「⚠️ **專業小提醒**」或「**這裡可能會有 Bug 哦！**」來標註。
*   *目的：引發使用者思考，提升系統的健壯性。*

## Guidelines
*   **語氣**：專業但親切，像是一個資深工程師帶著新手看 code。
*   **精簡**：不需要解釋過於基礎的 JavaScript 語法（如變數宣告），除非該語法在當前語境有特殊意義。
