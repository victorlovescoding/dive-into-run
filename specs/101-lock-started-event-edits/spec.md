# Feature Specification: 活動開始後鎖定編輯與刪除

**Feature Branch**: `101-lock-started-event-edits`
**Created**: 2026-06-12
**Status**: Draft
**Input**: User description: "活動目前只有開始時間；MVP 不新增或推算其他時間點。當可信系統時間達到活動開始時間後，鎖定主揪對活動本體的編輯與刪除，並保留不受影響的互動功能。"

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 主揪在活動開始後不能編輯活動本體 (Priority: P1)

主揪可以在活動開始前維持既有編輯能力；一旦活動開始當下或之後，活動本體資訊進入鎖定狀態，編輯入口保留但不可操作，並清楚說明原因，避免參加者已看到並依賴的活動資訊被靜默改寫。

**Why this priority**: 這是本需求的核心保護；活動開始後，時間、地點、路線、名額、備註等資訊已被參加者依賴，不應再被主揪改動。

**Independent Test**: 以主揪身份分別查看開始前與已開始的活動，確認開始前可編輯、開始後編輯入口不可操作且顯示「活動已開始，無法編輯或刪除。」；再驗證開始前開啟表單但開始後送出時，活動資料不會更新。

**Acceptance Scenarios**:

1. **Given** 使用者是活動主揪且可信系統時間早於活動開始時間，**When** 使用者查看活動編輯入口，**Then** 編輯入口維持既有可操作狀態。
2. **Given** 使用者是活動主揪且可信系統時間等於活動開始時間，**When** 使用者查看活動編輯入口，**Then** 編輯入口保留但呈現 disabled 狀態。
3. **Given** 使用者是活動主揪且可信系統時間晚於活動開始時間，**When** 使用者查看活動編輯入口，**Then** 編輯入口保留但呈現 disabled 狀態。
4. **Given** 活動編輯入口因活動已開始而 disabled，**When** 主揪查看或嘗試操作該入口，**Then** 系統顯示原因「活動已開始，無法編輯或刪除。」。
5. **Given** 主揪在活動開始前開啟編輯表單，**When** 活動開始後才送出儲存，**Then** 儲存失敗，活動資料不可部分更新。
6. **Given** 主揪嘗試編輯已開始活動的任一活動本體欄位，**When** 系統判斷可信系統時間已達活動開始時間，**Then** 編輯被拒絕且活動本體維持原狀。
7. **Given** 主揪編輯尚未開始的活動，**When** 將新的活動開始時間改成現在或過去，**Then** 系統阻止儲存，新的活動開始時間必須仍在未來。

---

### User Story 2 - 主揪在活動開始後不能刪除活動 (Priority: P1)

主揪可以在活動開始前維持既有刪除能力；一旦活動開始當下或之後，活動不可被刪除。刪除入口保留但不可操作，並與編輯使用相同的原因說明。

**Why this priority**: 刪除活動會直接破壞參加者已依賴的活動資訊與脈絡；開始後必須與編輯同等鎖定。

**Independent Test**: 以主揪身份分別查看開始前與已開始的活動，確認開始前可進入刪除確認，開始後刪除入口 disabled；再驗證開始前開啟刪除確認但開始後才確認時，活動仍存在。

**Acceptance Scenarios**:

1. **Given** 使用者是活動主揪且可信系統時間早於活動開始時間，**When** 使用者查看活動刪除入口，**Then** 刪除入口維持既有可操作狀態。
2. **Given** 使用者是活動主揪且可信系統時間等於活動開始時間，**When** 使用者查看活動刪除入口，**Then** 刪除入口保留但呈現 disabled 狀態。
3. **Given** 使用者是活動主揪且可信系統時間晚於活動開始時間，**When** 使用者查看活動刪除入口，**Then** 刪除入口保留但呈現 disabled 狀態。
4. **Given** 活動刪除入口因活動已開始而 disabled，**When** 主揪查看或嘗試操作該入口，**Then** 系統顯示原因「活動已開始，無法編輯或刪除。」。
5. **Given** 主揪在活動開始前開啟刪除確認，**When** 活動開始後才確認刪除，**Then** 刪除失敗，活動仍存在。
6. **Given** 主揪透過舊畫面、直接操作、重送請求、快速連點或多裝置操作嘗試刪除已開始活動，**When** 系統判斷可信系統時間已達活動開始時間，**Then** 刪除被一致拒絕且活動仍存在。

---

### User Story 3 - 非主揪任何時間都不能編輯或刪除活動 (Priority: P1)

非主揪不論活動是否開始，都不可編輯或刪除活動。若同時符合非主揪與活動已開始，對使用者呈現的主要原因應優先是沒有權限，而不是活動已開始。

**Why this priority**: 權限邊界是活動管理的基礎；開始時間鎖定不能削弱既有非主揪限制，也不能讓錯誤原因混淆。

**Independent Test**: 以非主揪身份在活動開始前、開始當下與開始後嘗試編輯或刪除，確認都被拒絕，且主要錯誤原因為沒有權限。

**Acceptance Scenarios**:

1. **Given** 使用者不是活動主揪且可信系統時間早於活動開始時間，**When** 使用者嘗試編輯活動，**Then** 編輯被拒絕，主要原因是沒有權限。
2. **Given** 使用者不是活動主揪且可信系統時間等於活動開始時間，**When** 使用者嘗試編輯活動，**Then** 編輯被拒絕，主要原因是沒有權限。
3. **Given** 使用者不是活動主揪且可信系統時間晚於活動開始時間，**When** 使用者嘗試編輯活動，**Then** 編輯被拒絕，主要原因是沒有權限。
4. **Given** 使用者不是活動主揪且可信系統時間早於活動開始時間，**When** 使用者嘗試刪除活動，**Then** 刪除被拒絕，主要原因是沒有權限。
5. **Given** 使用者不是活動主揪且可信系統時間等於或晚於活動開始時間，**When** 使用者直接嘗試刪除活動，**Then** 刪除被拒絕，主要原因是沒有權限。

---

### User Story 4 - 活動開始後其他活動互動不受鎖定影響 (Priority: P2)

活動開始後鎖定的範圍僅限活動本體編輯與刪除。使用者仍可查看活動詳情、留言、分享、收藏或取消收藏、參加或退出活動，除非既有規則本來就阻擋該操作。

**Why this priority**: 本功能目標是保護活動本體資訊，不是凍結整個活動頁；過度鎖定會破壞既有活動互動。

**Independent Test**: 在已開始活動上分別測試查看詳情、留言、分享、收藏、取消收藏、參加與退出，確認不會因本功能被阻擋；若既有規則阻擋，仍依既有規則處理。

**Acceptance Scenarios**:

1. **Given** 活動已開始，**When** 使用者查看活動詳情，**Then** 活動詳情仍可正常查看。
2. **Given** 活動已開始且使用者符合既有留言規則，**When** 使用者新增留言或刪除自己的留言，**Then** 操作不因本功能被阻擋。
3. **Given** 活動已開始，**When** 使用者分享活動連結，**Then** 分享不因本功能被阻擋。
4. **Given** 活動已開始，**When** 使用者收藏或取消收藏活動，**Then** 操作不因本功能被阻擋。
5. **Given** 活動已開始且使用者符合既有參加或退出規則，**When** 使用者參加或退出活動，**Then** 操作不因本功能被阻擋。
6. **Given** 活動已開始，**When** 系統顯示既有參加紀錄、留言、收藏紀錄、主揪或參加者身份，**Then** 這些資訊不因本功能改變。

### Edge Cases

- 活動開始後的判斷包含開始時間當下：可信系統時間等於活動開始時間時，活動本體已鎖定。
- 使用者裝置時間與可信系統時間不一致時，鎖定判斷以可信系統時間為準。
- 主揪在活動開始前已開啟編輯表單，但送出時活動已開始，儲存必須失敗且不得留下部分更新。
- 主揪在活動開始前已開啟刪除確認，但確認時活動已開始，刪除必須失敗且活動仍存在。
- 使用舊畫面、直接操作、重送請求、快速連點或多裝置操作修改已開始活動時，結果必須一致拒絕。
- 使用舊畫面、直接操作、重送請求、快速連點或多裝置操作刪除已開始活動時，結果必須一致拒絕。
- 使用者同時符合非主揪與活動已開始時，主要錯誤原因應優先是沒有權限。
- 編輯尚未開始的活動時，新的活動開始時間不得是現在或過去。
- 參加或退出可能影響人數統計，但不屬於本次活動本體編輯或刪除鎖定範圍。

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 系統 MUST 將「活動開始後」定義為可信系統時間大於或等於活動開始時間。
- **FR-002**: 系統 MUST 將「活動開始前」定義為可信系統時間早於活動開始時間。
- **FR-003**: 系統 MUST 在判斷活動本體是否鎖定時使用可信系統時間，不得以使用者裝置時間作為唯一依據。
- **FR-004**: 活動本體 MUST 包含主揪可編輯的活動核心資料，例如標題、開始時間、報名截止、地點、集合地點、路線、距離、配速、類型、名額與描述。
- **FR-005**: 主揪 MUST 能在活動開始前維持既有活動本體編輯能力。
- **FR-006**: 主揪 MUST 能在活動開始前維持既有活動刪除能力。
- **FR-007**: 主揪 MUST NOT 能在活動開始當下或之後編輯活動本體的任何欄位。
- **FR-008**: 主揪 MUST NOT 能在活動開始當下或之後刪除活動。
- **FR-009**: 非主揪 MUST NOT 在任何時間編輯活動。
- **FR-010**: 非主揪 MUST NOT 在任何時間刪除活動。
- **FR-011**: 活動開始後，系統 MUST 保留編輯入口與刪除入口，但兩者 MUST 呈現 disabled 狀態。
- **FR-012**: 活動開始後的 disabled 編輯與刪除入口 MUST 顯示清楚原因：「活動已開始，無法編輯或刪除。」。
- **FR-013**: 若主揪在活動開始前開啟編輯表單，但在活動開始後送出儲存，系統 MUST 拒絕儲存，且活動資料 MUST NOT 被部分更新。
- **FR-014**: 若主揪在活動開始前開啟刪除確認，但在活動開始後確認刪除，系統 MUST 拒絕刪除，且活動 MUST 仍存在。
- **FR-015**: 系統 MUST 對舊畫面、直接操作、重送請求、快速連點與多裝置操作嘗試修改已開始活動的情境維持一致拒絕結果。
- **FR-016**: 系統 MUST 對舊畫面、直接操作、重送請求、快速連點與多裝置操作嘗試刪除已開始活動的情境維持一致拒絕結果。
- **FR-017**: 編輯活動時，主揪 MUST NOT 能將新的活動開始時間設定為現在或過去；新的活動開始時間 MUST 仍在未來。
- **FR-018**: 若同一次嘗試同時符合非主揪與活動已開始，系統 MUST 將主要使用者可見原因優先呈現為沒有權限。
- **FR-019**: 刪除操作即使在底層行為不是永久移除，產品行為仍 MUST 視為刪除活動並受本鎖定規則限制。
- **FR-020**: 本功能 MUST NOT 新增其他時間欄位、推算額外時間點、取消狀態、公告、緊急通知、延期、改期流程、管理員覆寫、主揪申請解鎖、版本歷史或變更紀錄。
- **FR-021**: 本功能 MUST NOT 允許活動開始後修正任何活動本體欄位。
- **FR-022**: 本功能 MUST NOT 改變查看活動詳情、活動列表、搜尋、篩選、排序、留言、新增留言、刪除自己的留言、分享活動連結、收藏、取消收藏、參加、退出、既有參加紀錄、留言紀錄、收藏紀錄、主揪身份顯示或參加者身份顯示，除非既有規則本來就阻擋該操作。

### Key Entities

- **活動 (Event)**: 使用者建立並參與的跑步活動；包含活動開始時間、主揪、活動本體資訊與互動紀錄。活動開始後，活動本體進入編輯與刪除鎖定狀態。
- **活動本體**: 主揪可編輯的核心活動資料，例如標題、開始時間、報名截止、地點、集合地點、路線、距離、配速、類型、名額與描述；本次鎖定只作用於這個範圍及活動刪除。
- **主揪**: 建立或擁有活動管理權限的使用者；活動開始前可維持既有編輯與刪除能力，活動開始後不可編輯或刪除活動。
- **非主揪**: 不是活動主揪的使用者；任何時間都不可編輯或刪除活動。
- **可信系統時間**: 用於判斷活動是否已開始的時間來源；不得只依賴使用者裝置時間。

### Assumptions

- 本 spec 只處理活動本體編輯與刪除鎖定，不處理活動是否實際已在現場開始的人工判定。
- 活動目前只有活動開始時間；MVP 不新增其他時間欄位，也不估算任何額外時間點。
- 既有可編輯欄位集合即為本 spec 所稱的活動本體；若未來新增主揪可編輯的核心欄位，預設也受活動開始後鎖定規則保護。
- 參加與退出雖可能改變人數統計，但不視為本次鎖定的活動本體編輯。
- 既有權限規則仍然有效；本功能新增的是主揪在活動開始後的編輯與刪除限制。

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 在 100% 的主揪開始前驗收案例中，編輯與刪除入口維持既有可操作狀態。
- **SC-002**: 在 100% 的主揪開始當下與開始後驗收案例中，編輯與刪除入口保留但呈現 disabled 狀態。
- **SC-003**: 在 100% 的已開始活動編輯與刪除 disabled 驗收案例中，使用者可看到「活動已開始，無法編輯或刪除。」原因。
- **SC-004**: 在 100% 的開始前開啟編輯表單、開始後送出案例中，儲存被拒絕且活動資料沒有部分更新。
- **SC-005**: 在 100% 的開始前開啟刪除確認、開始後確認案例中，刪除被拒絕且活動仍存在。
- **SC-006**: 在 100% 的非主揪編輯或刪除驗收案例中，操作被拒絕，且主要錯誤原因為沒有權限。
- **SC-007**: 在 100% 的嘗試將活動開始時間改成現在或過去的編輯案例中，儲存被阻止。
- **SC-008**: 在舊畫面、直接操作、重送請求、快速連點與多裝置操作的驗收案例中，已開始活動的編輯與刪除 100% 被一致拒絕。
- **SC-009**: 在已開始活動的留言、分享、收藏、取消收藏、參加與退出驗收案例中，符合既有規則的操作 100% 不因本功能被阻擋。
- **SC-010**: 在產品文字與驗收文件檢查中，功能語意 100% 使用「活動開始後」與活動開始時間，不引入其他時間欄位或推算額外時間點。
