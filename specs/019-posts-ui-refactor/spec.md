# Feature Specification: Posts 頁面 UI 重新設計

**Feature Branch**: `019-posts-ui-refactor`  
**Created**: 2026-04-15  
**Status**: Draft  
**Input**: User description: "依據 PRD-posts-ui-refactor.md，將 Posts 列表頁與詳文頁升級為社群風格的現代化介面"

## User Scenarios & Testing _(mandatory)_

### User Story 1 — Post Feed 卡片重新設計 (Priority: P1)

使用者開啟 Posts 列表頁時，看到的不再是藍色邊框的開發樣式卡片，而是具備社群感的現代化 feed。每張 Post 卡片以平坦分線風格呈現（小圓角 4-8px 或無圓角、細線分隔），整體為亮色系配色（白/淺色背景、深色文字）。每張卡片清楚顯示：作者頭像、顯示名稱、相對時間（如「3 小時前」）、文章標題、內容摘要、愛心按讚數與留言數。作者可見的編輯/刪除選單保持不變。

**Why this priority**: 卡片是 feed 的核心視覺單元，所有其他功能都建構在這個基礎上。沒有重新設計的卡片，其他 UI 改善毫無意義。

**Independent Test**: 開啟 Posts 列表頁，確認每張卡片都以新的社群風格呈現，包含所有必要資訊欄位，且現有的編輯/刪除功能正常運作。

**Acceptance Scenarios**:

1. **Given** 列表頁有多篇文章，**When** 使用者開啟頁面，**Then** 每張卡片以平坦分線風格顯示，包含作者頭像、顯示名稱、相對時間、標題、內容摘要、按讚數與留言數
2. **Given** 使用者在桌面瀏覽器，**When** 縮小視窗至手機寬度，**Then** 卡片自適應調整版面，資訊仍完整可讀
3. **Given** 使用者為文章作者，**When** 查看自己的文章卡片，**Then** 編輯/刪除選單可見且功能正常
4. **Given** 列表頁存在多篇文章，**When** 使用者向下滾動，**Then** 無限滾動機制正常載入更多卡片，底部載入指示配合新視覺風格

---

### User Story 2 — 內容展開式摘要 (Priority: P2)

文章內容超過 150 字時，卡片只顯示截斷後的摘要，截斷處顯示「......」，後方附一個「查看更多」按鈕。使用者點擊「查看更多」後，文章就地展開顯示完整內容，展開後為最終狀態（不提供收起按鈕）。未超過 150 字的文章直接顯示完整內容。

**Why this priority**: 內容摘要機制直接影響 feed 的可瀏覽性與閱讀體驗，是社群 feed 的關鍵互動模式。

**Independent Test**: 建立一篇超過 150 字的文章與一篇未超過 150 字的文章，確認截斷、展開行為符合預期。

**Acceptance Scenarios**:

1. **Given** 一篇內容超過 150 字的文章，**When** 在列表頁顯示，**Then** 內容截斷至 150 字，末尾顯示「......」與「查看更多」按鈕
2. **Given** 內容已截斷的卡片，**When** 使用者點擊「查看更多」，**Then** 文章就地展開顯示完整內容，「查看更多」按鈕消失
3. **Given** 一篇內容未超過 150 字的文章，**When** 在列表頁顯示，**Then** 直接顯示完整內容，無截斷標記或「查看更多」按鈕
4. **Given** 已展開的文章，**When** 使用者繼續瀏覽，**Then** 展開狀態維持不變，無「收起」按鈕

---

### User Story 3 — Feed 頂部發文入口與 Modal (Priority: P3)

登入使用者在 feed 最上方看到一個內嵌的「假輸入框」區塊，視覺上融入 feed 成為第一張卡片，包含使用者頭像與 placeholder 文字（如「分享你的跑步故事...」）。點擊後開啟居中的 Modal 彈窗，包含標題輸入、內容輸入、發布按鈕。發布後 Modal 關閉，新文章出現在 feed 最上方。未登入使用者看不到此區塊。

**Why this priority**: 發文入口是社群 feed 的重要互動元素，但不影響閱讀體驗。即使沒有新的發文入口，使用者仍可透過既有方式發文。

**Independent Test**: 以登入/未登入身分各開啟一次列表頁，確認假輸入框的顯示/隱藏，並完成一次從點擊到發布的完整流程。

**Acceptance Scenarios**:

1. **Given** 使用者已登入，**When** 開啟 Posts 列表頁，**Then** feed 最上方顯示包含使用者頭像的假輸入框區塊
2. **Given** 使用者未登入，**When** 開啟 Posts 列表頁，**Then** 不顯示假輸入框區塊
3. **Given** 使用者已登入，**When** 點擊假輸入框，**Then** 開啟居中 Modal，包含標題欄位、內容欄位、發布按鈕
4. **Given** Modal 已開啟且表單填寫完成，**When** 使用者點擊發布，**Then** Modal 關閉，新文章出現在 feed 最上方
5. **Given** 假輸入框區塊，**When** 使用者向下滾動，**Then** 區塊隨頁面捲動離開視窗（非固定定位）

---

### User Story 4 — Post 詳文頁與留言卡片 (Priority: P4)

使用者進入 Post 詳文頁時，文章區域以與列表頁同一套視覺風格呈現，顯示完整文章內容（不截斷）。留言區每則留言以獨立卡片包裝，每張留言卡片包含作者頭像、名稱、留言內容、作者操作選單。留言輸入區域配合新視覺風格。所有現有功能（按讚、分享、編輯/刪除、留言無限滾動、留言高亮）保持正常。

**Why this priority**: 詳文頁是閱讀體驗的延伸，但使用頻率低於列表頁，且 PRD 明確指出與列表頁同風格，設計複雜度較低。

**Independent Test**: 開啟任一文章的詳文頁，確認視覺風格一致、留言以獨立卡片呈現、所有互動功能正常。

**Acceptance Scenarios**:

1. **Given** 使用者點擊列表頁某篇文章，**When** 進入詳文頁，**Then** 文章區域以社群風格呈現完整內容，視覺風格與列表頁一致
2. **Given** 詳文頁有多則留言，**When** 使用者查看留言區，**Then** 每則留言以獨立卡片呈現，包含作者頭像、名稱、留言內容
3. **Given** 使用者為留言作者，**When** 查看自己的留言卡片，**Then** 編輯/刪除選單可見且功能正常
4. **Given** 詳文頁，**When** 使用者操作按讚、分享、留言功能，**Then** 所有功能正常運作，toast 通知正常顯示
5. **Given** 詳文頁有大量留言，**When** 使用者向下滾動，**Then** 留言無限滾動正常載入更多留言卡片

---

### User Story 5 — 骨架屏載入狀態 (Priority: P5)

頁面載入中或載入更多內容時，顯示灰色佔位方塊動畫（骨架屏），替代原本的空白或簡單 loading 指示。骨架屏的佈局應模擬實際卡片結構，讓使用者預期即將呈現的內容。

**Why this priority**: 骨架屏提升感知效能與載入體驗，但不影響核心功能，屬於體驗層面的錦上添花。

**Independent Test**: 在網路限速條件下開啟列表頁或詳文頁，確認骨架屏正確顯示並在內容載入後消失。

**Acceptance Scenarios**:

1. **Given** Posts 列表頁正在載入，**When** 使用者開啟頁面，**Then** 顯示模擬卡片結構的骨架屏動畫
2. **Given** 使用者向下滾動觸發無限載入，**When** 新內容載入中，**Then** 底部顯示骨架屏佔位
3. **Given** 骨架屏正在顯示，**When** 實際內容載入完成，**Then** 骨架屏平順地被實際內容取代
4. **Given** Post 詳文頁正在載入，**When** 使用者開啟頁面，**Then** 顯示文章區域與留言區域的骨架屏

---

### Edge Cases

- 文章內容剛好 150 字時：直接顯示完整內容，不截斷
- 文章內容為 151 字時：截斷至 150 字並顯示「......」與「查看更多」
- 文章無內容（僅標題）：卡片正常顯示標題，內容區域為空或不顯示
- 作者無頭像：顯示預設頭像佔位
- 作者顯示名稱過長：適當截斷或以省略號處理，不破壞版面
- 列表頁無文章：顯示空狀態提示（如「還沒有文章，來寫第一篇吧！」）
- 網路斷線時載入更多：顯示適當的錯誤提示，保留已載入的內容
- Modal 中按 Escape 或點擊外部區域：表單空白時直接關閉；表單有內容時不關閉（防止誤觸遺失編輯）

## Requirements _(mandatory)_

### Functional Requirements

**列表頁 — 卡片與 Feed**

- **FR-001**: 系統 MUST 以平坦分線風格（圓角 8px、細線分隔）呈現每張 Post 卡片
- **FR-002**: 每張 Post 卡片 MUST 顯示：作者頭像、顯示名稱、相對時間、文章標題、內容摘要、按讚數、留言數
- **FR-003**: 整體頁面 MUST 採用亮色系配色（白/淺色背景、深色文字）
- **FR-004**: 卡片版面 MUST 支援響應式設計，在桌面與行動裝置皆正常顯示
- **FR-004a**: Feed 在桌面端 MUST 以置中窄欄呈現（max-width 約 640-680px），不得全寬延伸

**列表頁 — 內容摘要**

- **FR-005**: 文章內容超過 150 字時，系統 MUST 截斷內容並在截斷處顯示「......」
- **FR-006**: 截斷內容的「......」後方 MUST 顯示「查看更多」按鈕
- **FR-007**: 點擊「查看更多」MUST 以平滑過渡動畫（高度漸變，約 200-300ms）就地展開顯示完整文章內容
- **FR-008**: 展開後的文章 MUST NOT 顯示「收起」按鈕

**列表頁 — 發文入口**

- **FR-009**: 登入使用者的 feed 頂部 MUST 顯示內嵌假輸入框區塊，包含使用者頭像與 placeholder 文字
- **FR-010**: 未登入使用者 MUST NOT 看到假輸入框區塊
- **FR-011**: 假輸入框 MUST 隨頁面滾動（非固定定位）
- **FR-012**: 點擊假輸入框 MUST 開啟居中 Modal 彈窗
- **FR-013**: 發文 Modal MUST 包含標題輸入、內容輸入、發布按鈕
- **FR-014**: 發布成功後 Modal MUST 關閉，新文章 MUST 出現在 feed 最上方

**詳文頁**

- **FR-015**: 詳文頁文章區域 MUST 與列表頁採用同一套視覺風格
- **FR-016**: 詳文頁 MUST 顯示完整文章內容，不截斷
- **FR-017**: 每則留言 MUST 以獨立卡片呈現，包含作者頭像、名稱、留言內容
- **FR-018**: 留言作者 MUST 可見編輯/刪除操作選單
- **FR-019**: 留言輸入區域 MUST 配合新視覺風格

**骨架屏**

- **FR-020**: 頁面載入中 MUST 顯示模擬實際卡片結構的灰色骨架屏動畫
- **FR-021**: 無限滾動載入更多內容時 MUST 顯示骨架屏佔位
- **FR-022**: 內容載入完成後骨架屏 MUST 被實際內容取代

**功能保留**

- **FR-023**: 所有現有功能 MUST 保持正常運作，包含：文章 CRUD、按讚、無限滾動、toast 通知、留言高亮、留言 CRUD
- **FR-024**: 系統 MUST NOT 新增任何功能邏輯，本次為純 UI/UX 重構

## Clarifications

### Session 2026-04-15

- Q: Feed 在桌面端（1280px+）的排版策略？ → A: 置中窄欄，feed max-width 約 640-680px，居中顯示
- Q: 卡片圓角值？ → A: 8px（標準社群圓角）
- Q:「查看更多」展開過渡效果？ → A: 平滑過渡動畫（高度漸變，約 200-300ms）

## Assumptions

- 「150 字」指 Unicode 字元數（含中文字），非 byte 數
- 相對時間格式（如「3 小時前」）沿用現有實作邏輯，不另外定義格式規則
- 假輸入框的 placeholder 文字為「分享你的跑步故事...」，可視設計需求微調
- 預設頭像為系統既有的 fallback 機制
- Modal 的關閉行為（Escape / 點擊外部）沿用現有互動慣例
- 骨架屏的動畫形式為常見的 shimmer/pulse 效果

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 使用者開啟 Posts 列表頁時，所有卡片以社群風格呈現，不殘留任何舊版開發樣式（藍色邊框、紅色邊框底部、固定右下角 ➕ 按鈕）
- **SC-002**: 100% 的現有功能（文章 CRUD、按讚、留言 CRUD、無限滾動、toast 通知、留言高亮）在重構後仍正常運作
- **SC-003**: 頁面在桌面（1280px+）與行動裝置（375px）寬度下皆正確顯示，無破版或資訊遺漏
- **SC-004**: 超過 150 字的文章內容在列表頁正確截斷，點擊「查看更多」可就地展開
- **SC-005**: 頁面載入中顯示骨架屏，使用者感知到的載入等待體驗優於重構前的空白/spinner 狀態
- **SC-006**: 登入使用者可透過 feed 頂部假輸入框完成發文流程（點擊→填寫→發布→文章出現在 feed）
