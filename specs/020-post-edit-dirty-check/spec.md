# Feature Specification: Post Edit Dirty Check

**Feature Branch**: `020-post-edit-dirty-check`
**Created**: 2026-04-16
**Status**: Draft
**Input**: User description: "我要修改文章頁面的功能，編輯文章如果完全沒有做更動不可以變更，要跟活動頁面的編輯活動一樣邏輯"

## Clarifications

### Session 2026-04-16

- Q: 使用者點擊「更新」後、請求回來前的按鈕狀態？ → A: 按鈕停用，文字從「更新」切換為「更新中…」（對齊活動頁 `disabled={!isDirty || !!isSubmitting}` 慣例）。
- Q: 使用者把標題或內文改成「全空白」（trim 後為空字串）時，dirty 與 validation 的 gate 組合？ → A: dirty 成立即讓按鈕亮；按下後由 `validatePostInput` 擋下並顯示 error toast，對話框保持開啟。沿用現況 lazy validation，兩個 gate 解耦。

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 阻擋無變更的文章編輯送出 (Priority: P1)

作為文章作者，當我打開「編輯文章」表單但最終沒有修改任何欄位時，系統不應允許我觸發「更新」動作，避免白做一次無意義的儲存並污染文章的「最後編輯時間」等衍生資訊。行為需與目前活動（Event）編輯頁完全一致——送出按鈕必須在「完全沒有任何變更」時保持停用狀態。

**Why this priority**: 這是唯一的核心訴求；沒這段就等於這個 feature 不存在。

**Independent Test**: 登入後進入自己的文章，點選編輯，在不修改任何欄位的情況下檢查送出按鈕是否呈停用狀態且無法觸發更新；另驗證修改後再改回原樣時按鈕同樣回到停用狀態。

**Acceptance Scenarios**:

1. **Given** 使用者在文章詳情頁打開自己文章的「編輯文章」對話框，**When** 未修改任何欄位，**Then** 「更新」按鈕呈停用狀態，使用者無法送出，也不會產生任何後端寫入或使用者提示訊息。
2. **Given** 使用者在文章列表頁打開自己文章的「編輯文章」對話框，**When** 未修改任何欄位，**Then** 「更新」按鈕呈停用狀態，使用者無法送出。
3. **Given** 使用者已修改標題或內文，**When** 將所有欄位值改回與原始內容完全相同，**Then** 「更新」按鈕回到停用狀態。
4. **Given** 使用者已修改標題或內文並讓按鈕成為可用狀態，**When** 點擊「更新」，**Then** 系統執行更新並顯示既有的成功訊息、關閉對話框、列表/詳情頁顯示最新內容。
5. **Given** 使用者打開編輯對話框，**When** 僅在原標題或原內文的前端或尾端加入空白字元（前後 trim 後與原始值相同），**Then** 「更新」按鈕維持停用狀態。
6. **Given** 使用者修改標題內容且確實產生差異（例如改動了中間的文字），**When** 點擊「更新」，**Then** 系統存入的是「前後 trim 後」的標題/內文，中間空白完整保留。

---

### User Story 2 - 新增文章不受此規則影響 (Priority: P2)

作為使用者，當我建立一篇**新**文章時，不應受到「沒有變更就不能送出」這條規則影響——新增模式本來就沒有「原始值」可以比較，其送出條件應維持現行行為（有合法輸入即可送出）。

**Why this priority**: 避免把新增流程意外鎖住；雖然風險低，仍需明確界定。

**Independent Test**: 在文章列表頁點擊「發表文章」，確認「發布」按鈕在輸入合法標題/內容後可正常送出，且流程與現況一致。

**Acceptance Scenarios**:

1. **Given** 使用者開啟「發表文章」對話框（非編輯模式），**When** 輸入合法內容，**Then** 「發布」按鈕可正常點擊並送出。
2. **Given** 使用者開啟「發表文章」對話框，**When** 未輸入任何內容，**Then** 行為維持既有驗證邏輯（非本 feature 範疇）。

---

### Edge Cases

- **多次開關編輯對話框**：關閉未送出的編輯再重新開啟，dirty 比較的「原始值」基準點必須以當下資料庫中的最新內容為準，不能殘留上一次 session 的狀態。
- **編輯過程中原文在別處被更新**：若使用者開啟編輯後、另一個 session 更新了該文章，使用者最終送出時以當時記住的原始值為 dirty 比較基準（與活動頁相同策略；不在此 feature 範圍內處理樂觀鎖）。
- **前後空白字元**：系統對標題與內文值採「前後 trim」策略進行 dirty 比較與實際寫入。因此「原始內容 + 僅尾端多幾個空白」視為**無變更**（按鈕維持停用），且真的送出時也會把前後空白清掉後再存。中間空白（詞與詞之間、段落換行、內文空行）完全保留，避免破壞排版。
- **僅修改標題** 或 **僅修改內文**：任一欄位有差異即視為 dirty，按鈕應可點擊。
- **改成全空白（trim 後為空字串）**：標題或內文被改為純空白時，dirty 判定成立（按鈕亮起），但送出時由 `validatePostInput` 攔截並顯示 error toast，對話框保持開啟以便使用者修正。亦即 dirty 和 validation 為兩道獨立的 gate：dirty 只管「有沒有改」、validation 於送出時才檢查「改得合不合法」。
- **送出過程中**：點擊「更新」後、請求尚未回來前，按鈕 MUST 呈停用狀態且文字切換為「更新中…」（對齊活動頁慣例），避免重複送出並給使用者明確的處理中回饋。

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 在文章編輯模式下，系統 MUST 記住開啟編輯時的原始標題與原始內文，作為 dirty 比較基準。
- **FR-002**: 系統 MUST 在「編輯文章」對話框中，依據「當前輸入值 vs 原始值」的逐欄位比較結果計算是否 dirty；任一欄位不同即為 dirty。
- **FR-003**: 當編輯模式下表單為非 dirty 狀態，「更新」按鈕 MUST 呈停用狀態，且不可透過鍵盤或其他方式觸發送出。
- **FR-004**: 當使用者修改後又將所有欄位改回原始值，系統 MUST 偵測到恢復為非 dirty 狀態並重新停用「更新」按鈕。
- **FR-005**: 新增文章流程 MUST 不受此 dirty 規則影響，其送出條件維持現行行為。
- **FR-006**: 此規則 MUST 同時套用於文章列表頁（`/posts`）與文章詳情頁（`/posts/[id]`）兩處的編輯入口，行為與使用者感受一致。
- **FR-007**: 在非 dirty 狀態下嘗試送出 MUST 不產生任何後端寫入、不顯示成功/失敗通知、不關閉對話框。
- **FR-008**: 在 dirty 狀態下成功送出後，按鈕 MUST 自動回到停用狀態（因為送出後的內容即為新的「原始值」，或對話框已關閉），避免立即重複送出同樣內容。送出進行中時（請求已發出、尚未回應），按鈕 MUST 停用且文字切換為「更新中…」。
- **FR-009**: 整體使用者體驗（送出按鈕文字、禁用外觀、對話框開關動線）MUST 對齊活動編輯頁已建立的慣例，保持跨功能的一致性。
- **FR-010**: 系統 MUST 對標題與內文的「前後空白字元」做 trim 處理；dirty 比較與實際送出寫入皆使用相同的 trim 結果，確保「按鈕狀態」與「真正寫入資料庫的值」永遠一致，不會出現判定與實際寫入分叉的情況。中間空白（詞間、段落換行、內文空行）MUST 完整保留。

### Key Entities

- **Post Draft State**：使用者在編輯對話框內的暫存編輯狀態，包含當前輸入的標題、內文；需與「原始值快照」比較以判定 dirty。
- **Original Post Snapshot**：對話框開啟當下所載入的文章原始標題與內文，為 dirty 比較的基準；在對話框單次生命週期內保持不變。

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 開啟「編輯文章」後不做任何修改，100% 的嘗試送出行為都必須被阻擋（按鈕停用）。
- **SC-002**: 修改任一欄位再改回原樣的情境下，送出按鈕 100% 回到停用狀態。
- **SC-003**: 針對「沒有變更的無意義編輯」所產生的後端寫入次數下降至 0。
- **SC-004**: 同一位使用者在文章列表頁與文章詳情頁進行「編輯文章」時，感知到的「送出按鈕啟用條件」表現 100% 一致，也與活動編輯頁表現一致。

## Assumptions

- 文章目前僅有「標題」與「內文」兩個可編輯欄位（來源：`updatePost(id, { title, content })` 的既有介面）；若未來新增可編輯欄位，dirty 比較策略需同步擴充。
- 標題與內文採「前後 trim」策略：dirty 比較與實際送出寫入皆以 `trim` 後的值為準，保證按鈕狀態與實際寫入資料一致。中間空白（詞間、換行、段落空行）完整保留，避免破壞內文排版。此策略與目前活動編輯頁略有差異（活動頁目前採逐字比較）；決策理由：文章輸入較常出現 copy-paste 帶入的無意義前後空白，清理能有效避免誤判與髒資料。
- 文章編輯入口僅在「文章列表頁」與「文章詳情頁」兩處，皆透過共用的編輯對話框開啟。
- 此 feature 僅調整「送出閘門」行為，不改變現有的成功/失敗通知、Firestore schema、或列表/詳情頁的資料更新邏輯。
- 現行的輸入驗證（`validatePostInput`）採 lazy validation：於「按下送出」時才檢查，非於輸入過程即時檢查。dirty（button enable 門檻）與 validation（send 門檻）為兩道獨立 gate，AND 關係意指「兩個都要過才真的送出」，而非「兩個都要過才亮按鈕」。此設計延續既有 code，避免此 feature scope 擴張到改動驗證時機。
