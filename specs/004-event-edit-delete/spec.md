# Feature Specification: Event Edit & Delete Actions

**Feature Branch**: `004-event-edit-delete`
**Created**: 2026-03-29
**Status**: Draft
**Input**: User description: "在 event 頁面的每個活動卡片右上角新增三個點按鈕（僅活動創建人可見），提供編輯活動和刪除活動功能。編輯復用現有建立表單並預填資料，刪除需確認後從 Firestore 移除。"

## Clarifications

### Session 2026-03-29

- Q: 編輯人數上限時，若目前報名人數已超過新設定的上限，系統應如何處理？ → A: 限制人數上限不能低於目前報名人數（例如已有 8 人報名，最小值為 8）

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 活動創建人編輯自己的活動 (Priority: P1)

活動創建人在活動列表頁看到自己建立的活動卡片右上角有一個「三個點」按鈕，點擊後出現下拉選單，選擇「編輯活動」後彈出表單，表單內已預填該活動的所有現有資料。創建人修改需要更改的欄位後，點擊「編輯完成」按鈕送出更新。

**Why this priority**: 編輯是最核心的功能，讓活動發起人可以修正錯誤或更新活動資訊，直接影響活動品質與參與者體驗。

**Independent Test**: 登入為活動創建人，在活動列表點擊三個點 > 編輯活動，確認表單預填正確資料，修改任一欄位後點擊「編輯完成」，驗證活動資料已更新。

**Acceptance Scenarios**:

1. **Given** 使用者已登入且為某活動的創建人, **When** 使用者在活動列表頁查看該活動卡片, **Then** 卡片右上角顯示三個點按鈕
2. **Given** 使用者已登入但不是某活動的創建人, **When** 使用者在活動列表頁查看該活動卡片, **Then** 卡片右上角不顯示三個點按鈕
3. **Given** 使用者點擊三個點按鈕, **When** 下拉選單出現, **Then** 顯示「編輯活動」和「刪除活動」兩個選項
4. **Given** 使用者點擊「編輯活動」, **When** 編輯表單彈出, **Then** 表單內所有欄位已預填該活動的現有資料（標題、時間、截止時間、地點、集合點、跑步類型、距離、配速、人數上限、描述等）
5. **Given** 編輯表單已開啟且尚未修改任何欄位, **When** 使用者查看「編輯完成」按鈕, **Then** 按鈕顯示為不可點擊狀態（視覺上以顏色區分）
6. **Given** 編輯表單已開啟且使用者已修改至少一個欄位, **When** 使用者查看「編輯完成」按鈕, **Then** 按鈕變為可點擊狀態（視覺上以顏色區分）
7. **Given** 使用者在編輯表單修改了欄位並點擊「編輯完成」, **When** 更新成功, **Then** 表單關閉，活動列表中該活動卡片顯示更新後的資料
8. **Given** 使用者在編輯表單點擊「取消編輯」, **When** 表單關閉, **Then** 活動資料不受影響

### User Story 2 - 活動創建人刪除自己的活動 (Priority: P2)

活動創建人在活動列表頁點擊自己活動卡片右上角的三個點按鈕，選擇「刪除活動」後，系統彈出自訂樣式的確認視窗（非瀏覽器原生 confirm）詢問「確定要刪除活動？」，確認後系統從資料庫刪除該活動並顯示結果訊息。

**Why this priority**: 刪除是必要但較低頻的操作，且具有破壞性，需要確認機制保護。

**Independent Test**: 登入為活動創建人，點擊三個點 > 刪除活動，確認彈出確認視窗，選擇「是」後驗證活動已從列表和資料庫中移除。

**Acceptance Scenarios**:

1. **Given** 使用者點擊「刪除活動」, **When** 確認視窗彈出, **Then** 顯示「確定要刪除活動？」訊息，並提供「是」和「否」兩個選項
2. **Given** 確認視窗已彈出, **When** 使用者選擇「否」, **Then** 確認視窗關閉，活動不受影響
3. **Given** 確認視窗已彈出, **When** 使用者選擇「是」且刪除成功, **Then** 顯示「刪除成功」訊息，活動從列表中移除
4. **Given** 確認視窗已彈出, **When** 使用者選擇「是」但刪除失敗, **Then** 顯示「發生錯誤，請再試一次」訊息，活動保持不變

### User Story 3 - 編輯表單的按鈕行為與視覺回饋 (Priority: P3)

編輯表單的底部按鈕與建立表單有所區別：左邊按鈕名稱為「取消編輯」（而非「取消」），右邊按鈕名稱為「編輯完成」（而非「建立活動」）。「編輯完成」按鈕僅在表單有任何資訊變更時才可點擊，並透過按鈕顏色的變化讓使用者清楚知道目前是否有未儲存的變更。

**Why this priority**: 視覺回饋增強使用體驗，但不影響核心功能運作。

**Independent Test**: 開啟編輯表單，確認按鈕名稱正確，未修改時「編輯完成」不可按，修改任一欄位後按鈕變為可按狀態。

**Acceptance Scenarios**:

1. **Given** 編輯表單已開啟, **When** 使用者查看底部按鈕, **Then** 左邊按鈕顯示「取消編輯」，右邊按鈕顯示「編輯完成」
2. **Given** 表單欄位尚未被修改, **When** 使用者查看「編輯完成」按鈕, **Then** 按鈕以淡化顏色（disabled 狀態）呈現且不可點擊
3. **Given** 使用者修改了某欄位後又改回原值, **When** 所有欄位值與原始值完全一致, **Then**「編輯完成」按鈕回到不可點擊狀態

---

### Edge Cases

- 使用者在編輯表單開啟期間，活動已被其他管道刪除或修改？以使用者提交時的結果為準，若活動不存在則顯示錯誤訊息。
- 使用者同時開啟多個活動的編輯表單？同一時間只能開啟一個編輯表單，開啟新的會關閉舊的。
- 點擊三個點按鈕後，點擊選單外部區域？選單自動關閉。
- 刪除的活動已有參與者？仍然允許刪除，一併移除相關的參與者資料。
- 未登入使用者查看活動列表？三個點按鈕不顯示（因為沒有登入使用者，不可能是任何活動的創建人）。
- 編輯人數上限低於目前報名人數？不允許，人數上限最小值為目前已報名人數。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統必須在每個活動卡片右上角顯示三個點按鈕，但僅限於該活動的創建人與當前登入使用者為同一人時才顯示
- **FR-002**: 點擊三個點按鈕後，系統必須顯示包含「編輯活動」和「刪除活動」兩個選項的下拉選單
- **FR-003**: 點擊選單外部區域時，下拉選單必須自動關閉
- **FR-004**: 選擇「編輯活動」時，系統必須開啟與建立活動相同的表單，並預填該活動的所有現有資料
- **FR-005**: 編輯表單底部左邊按鈕必須顯示「取消編輯」，右邊按鈕必須顯示「編輯完成」
- **FR-006**: 「編輯完成」按鈕必須在表單資料與原始資料完全一致時處於不可點擊狀態，且視覺上以顏色區分
- **FR-007**: 「編輯完成」按鈕必須在表單任何欄位值與原始資料不同時變為可點擊狀態，且視覺上以顏色區分
- **FR-008**: 點擊「編輯完成」後，系統必須將更新後的資料寫入資料庫，成功後關閉表單並更新活動列表中的對應卡片
- **FR-009**: 編輯失敗時，系統必須顯示錯誤訊息，表單保持開啟狀態讓使用者可以重試
- **FR-010**: 選擇「刪除活動」時，系統必須彈出自訂樣式的確認視窗（非瀏覽器原生 confirm），顯示「確定要刪除活動？」並提供「是」和「否」兩個按鈕
- **FR-011**: 確認刪除後，系統必須從資料庫刪除該活動文件及其相關的參與者子集合
- **FR-012**: 刪除成功後，系統必須顯示「刪除成功」訊息並從活動列表中移除該卡片
- **FR-013**: 刪除失敗時，系統必須顯示「發生錯誤，請再試一次」訊息，活動保持不變
- **FR-014**: 同一時間只能開啟一個活動的編輯表單或下拉選單
- **FR-015**: 編輯表單中「人數上限」欄位的最小值必須為目前已報名人數與 2 之間的較大者，不允許設定低於目前報名人數的值

### Key Entities

- **活動 (Event)**: 包含標題、時間、截止時間、地點、跑步類型、距離、配速、人數上限、描述、路線等屬性，以及創建人資訊。編輯操作修改此實體，刪除操作移除此實體。
- **參與者 (Participant)**: 活動的子集合，記錄已報名的使用者。刪除活動時需一併處理。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 活動創建人可以在 3 次點擊內完成編輯表單的開啟（點擊三個點 > 編輯活動）
- **SC-002**: 編輯表單預填的資料與原始活動資料 100% 一致
- **SC-003**: 使用者修改欄位後，「編輯完成」按鈕在 1 秒內反映狀態變化
- **SC-004**: 刪除操作在確認後 3 秒內完成並顯示結果訊息
- **SC-005**: 非活動創建人在任何情況下都看不到三個點按鈕
- **SC-006**: 活動編輯後，其他使用者重新載入頁面即可看到更新後的資料

## Assumptions

- 編輯活動不需要額外的權限驗證（前端以 hostUid 比對即可），Firestore Security Rules 已有相應保護
- ts 子集合中的所有文件刪除活動時一併刪除 participant
- 編輯表單復用現有建立表單的 UI 結構和驗證邏輯
- 路線資料（polyline/routeCoordinates）在編輯時可以重新繪製或保留原始值
- 表單的「已變更」偵測採用逐欄位值比較（shallow comparison）
