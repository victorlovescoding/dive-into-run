# Feature Specification: 使用者公開檔案頁面 (Public Profile)

**Feature Branch**: `012-public-profile`
**Created**: 2026-04-09
**Status**: Draft
**Input**: User description: "F-005 使用者公開檔案頁面 — 建立 /users/[uid] 頁面，顯示跑者的公開資訊、開團數、參團數"

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 瀏覽主揪的公開檔案 (Priority: P1)

身為揪團參加者，我在考慮是否要報名某個揪團活動時，想先看看主揪是誰、過去開過哪些團、有沒有跑步經驗。我希望能點進主揪的公開頁面，看到他的基本資料、跑步統計和過去主辦的活動列表，這樣我才能判斷是否要信任這個團。

**Why this priority**: 信任是揪團平台的核心問題。跑者在報名陌生人開的團之前，需要有管道了解主揪的背景和經歷。這是公開檔案最直接的使用場景，也是整個功能存在的根本原因。

**Independent Test**: 直接輸入某使用者的公開檔案 URL，驗證可以看到該使用者的頭像、名稱、簡介、加入日期、統計數據、以及主辦的活動列表。

**Acceptance Scenarios**:

1. **Given** 使用者造訪某個跑者的公開檔案頁面, **When** 頁面載入完成, **Then** 顯示該跑者的頭像、名稱、個人簡介、加入平台日期
2. **Given** 使用者造訪一位有主辦活動紀錄的跑者檔案, **When** 頁面載入完成, **Then** 顯示統計數據：開團數、參團數、累計跑步公里數
3. **Given** 使用者造訪一位有主辦活動紀錄的跑者檔案, **When** 查看活動列表區塊, **Then** 可以看到該跑者主辦的公開活動，按時間倒序排列
4. **Given** 使用者造訪一位沒有主辦活動的跑者檔案, **When** 頁面載入完成, **Then** 活動列表區塊顯示「尚無主辦活動」的空狀態提示
5. **Given** 使用者造訪一位沒有連結 Strava 的跑者檔案, **When** 頁面載入完成, **Then** 累計跑步公里數欄位完全不顯示

---

### User Story 2 - 從活動卡片和留言點擊進入使用者檔案 (Priority: P1)

身為平台使用者，我在瀏覽活動列表、活動詳情頁的參與者名單、或者留言區時，看到其他跑者的名字和頭像。我希望能直接點擊他們的名字或頭像，跳轉到他們的公開檔案頁面。

**Why this priority**: 如果公開檔案頁面存在但使用者找不到入口，功能就沒有意義。全站的使用者名稱/頭像可點擊是讓公開檔案被使用的關鍵，與 P1 的檔案頁面本身同等重要。

**Independent Test**: 在活動卡片、活動詳情頁的主揪/參與者、留言區的作者名稱上點擊，驗證可以跳轉到對應使用者的公開檔案頁面。

**Acceptance Scenarios**:

1. **Given** 使用者在活動列表頁瀏覽活動卡片, **When** 點擊活動卡片上的主揪名稱或頭像, **Then** 跳轉到該主揪的公開檔案頁面
2. **Given** 使用者在活動詳情頁查看參與者名單, **When** 點擊某位參與者的名稱或頭像, **Then** 跳轉到該參與者的公開檔案頁面
3. **Given** 使用者在留言區閱讀留言, **When** 點擊留言作者的名稱或頭像, **Then** 跳轉到該留言作者的公開檔案頁面
4. **Given** 使用者在文章列表或文章詳情頁, **When** 點擊作者名稱或頭像, **Then** 跳轉到該作者的公開檔案頁面

---

### User Story 3 - 編輯個人簡介 (Priority: P2)

身為已登入使用者，我想要在我的會員頁面新增或編輯個人簡介（Bio），讓其他跑者在瀏覽我的公開檔案時能更了解我是誰、我的跑步背景和興趣。

**Why this priority**: 公開檔案的內容豐富度直接影響信任建立效果。簡介是最能展現個人特色的欄位，但它是「錦上添花」而非核心功能 — 即使沒有簡介，頭像、名稱和統計數據已能提供基本信任依據。

**Independent Test**: 登入後進入會員頁面，新增或修改個人簡介文字，儲存後造訪自己的公開檔案頁面，驗證簡介內容正確顯示。

**Acceptance Scenarios**:

1. **Given** 已登入使用者在會員頁面, **When** 找到簡介編輯區塊, **Then** 可以輸入或修改個人簡介文字
2. **Given** 使用者輸入簡介文字並儲存, **When** 造訪自己的公開檔案頁面, **Then** 可以看到剛才輸入的簡介內容
3. **Given** 使用者尚未填寫簡介, **When** 造訪自己的公開檔案頁面, **Then** 簡介區塊完全不顯示
4. **Given** 使用者輸入超過字數上限(150字)的簡介, **When** 嘗試儲存, **Then** 顯示字數超出提示，無法儲存

---

### User Story 4 - 查看自己的公開檔案 (Priority: P3)

身為已登入使用者，我想要預覽自己的公開檔案，確認其他人看到的我的頁面是什麼樣子。

**Why this priority**: 讓使用者能確認自己的公開形象，是基本的使用者體驗需求，但優先度低於核心的瀏覽和連結功能。

**Independent Test**: 登入後透過導覽或直接輸入自己的 UID 公開檔案 URL，驗證可以看到自己的完整公開檔案，且有明確標示「這是你的公開檔案」。

**Acceptance Scenarios**:

1. **Given** 已登入使用者造訪自己的公開檔案, **When** 頁面載入完成, **Then** 顯示與其他人看到的相同內容，並有提示標示「這是你的公開檔案」
2. **Given** 已登入使用者在會員頁面, **When** 查看頁面, **Then** 可以看到「查看我的公開檔案」連結，點擊後跳轉到自己的公開檔案頁面
3. **Given** 已登入使用者在自己的公開檔案頁面, **When** 想要編輯資料, **Then** 有明確的連結或按鈕引導至會員頁面的編輯功能

---

### Edge Cases

- **不存在的使用者**：造訪不存在（或已刪除帳號）的使用者 UID 時，應顯示友善的「找不到此使用者」頁面，而非錯誤畫面
- **未登入使用者瀏覽**：未登入的訪客也應該能瀏覽公開檔案（公開檔案本質上是公開的）
- **主辦活動很多**：當使用者主辦了大量活動時，活動列表採用無限滾輪（infinite scroll）機制，每次載入 5 筆，避免一次載入過多資料
- **使用者無頭像**：沒有上傳頭像的使用者應顯示預設頭像
- **簡介中含特殊字元**：簡介內容需正確處理特殊字元和換行，防止 XSS 注入

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 系統 MUST 提供每位使用者一個可透過唯一 URL 存取的公開檔案頁面
- **FR-002**: 公開檔案頁面 MUST 顯示使用者的頭像、名稱、個人簡介、加入平台日期
- **FR-003**: 公開檔案頁面 MUST 顯示統計數據 — 主辦活動數量、參加活動數量、累計跑步公里數（來自已連結的跑步紀錄）
- **FR-004**: 公開檔案頁面 MUST 列出該使用者主辦的公開活動，按時間倒序排列，採用 infinite scroll 機制每次載入 5 筆
- **FR-005**: 平台上所有顯示使用者名稱或頭像的位置（活動卡片主揪、活動詳情主揪與參與者、留言作者、文章作者）MUST 變為可點擊的連結，導向該使用者的公開檔案頁面
- **FR-006**: 已登入使用者 MUST 能在會員頁面新增或編輯個人簡介，簡介字數上限為 150 字
- **FR-007**: 公開檔案頁面 MUST 可被未登入的訪客瀏覽（無須登入即可查看）
- **FR-008**: 當造訪不存在的使用者檔案時，系統 MUST 顯示友善的「找不到此使用者」提示頁面
- **FR-009**: 簡介內容 MUST 進行適當的輸出轉義，防止跨站腳本攻擊（XSS）

### Key Entities

- **公開使用者檔案 (Public User Profile)**: 代表一位使用者對外公開的個人資訊，包含頭像、名稱、簡介、加入日期。此為使用者私人帳號資料的公開子集
- **使用者統計 (User Statistics)**: 匯總一位使用者的平台活動數據，包含主辦活動數量、參加活動數量、累計跑步公里數。數據來源為平台既有的活動與跑步紀錄
- **主辦活動列表 (Hosted Events List)**: 該使用者以主揪身份建立的所有公開活動，按時間倒序呈現

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 使用者可在 2 秒內載入並瀏覽任何其他使用者的公開檔案頁面
- **SC-002**: 使用者可以從平台上任何出現使用者名稱/頭像的位置（活動卡片、活動詳情、留言、文章），一鍵點擊跳轉至對應的公開檔案
- **SC-003**: 使用者可在 1 分鐘內完成個人簡介的編輯和儲存
- **SC-004**: 公開檔案頁面在未登入狀態下仍可正常瀏覽，不出現權限錯誤
- **SC-005**: 不存在的使用者檔案頁面顯示友善提示，不出現技術性錯誤頁面

## Assumptions

- 所有使用者檔案預設為公開，v1 不提供隱私控制（關閉/隱藏檔案）
- 個人簡介（Bio）欄位為此功能新增，目前使用者資料中不包含此欄位
- 累計跑步公里數來自已連結 Strava 的跑步紀錄；未連結 Strava 的使用者不顯示此數據
- 活動列表僅顯示使用者以主揪身份建立的活動（不含參加的活動），與 roadmap 定義一致
- 簡介字數上限設為 150 字（純文字，不支援 Markdown 或富文字格式）

## Clarifications

### Session 2026-04-09

- Q: Bio 空狀態處理 — 不顯示 or 顯示預設提示文字？ → A: 不顯示，簡介區塊完全隱藏
- Q: 主辦活動列表每頁筆數？ → A: 5 筆
- Q: 未連結 Strava 時累計公里數顯示方式？ → A: 不顯示，欄位完全隱藏

### Session 2026-04-10

- Q: 活動列表每筆活動要顯示哪些資訊？ → A: 複用現有 EventCard 元件樣式，可點擊進入活動詳情
- Q: 活動列表要顯示哪些時間範圍？ → A: 全部（過去 + 未來的主辦活動都顯示）
- Q: 「參團數」怎麼定義？ → A: 報名即算（加入參與者名單就計入）
- Q: 查看自己公開檔案的入口放哪裡？ → A: 會員頁面放「查看我的公開檔案」連結
- Q: 連結 Strava 但累計 0 公里時怎麼處理？ → A: 顯示 `0 km`（有連結就顯示，不隱藏）
- Q: 公開檔案需要 SEO / Open Graph meta tags？ → A: 要做，用 Next.js generateMetadata，加 og:title / og:description / og:image
