# Feature Specification: 文章搜尋

**Feature Branch**: `103-posts-search`
**Created**: 2026-06-14
**Status**: Draft
**Input**: User description: "在 `/posts` 文章河道加入搜尋入口，並以獨立 `/posts/search?q={keyword}` 搜尋結果頁搜尋全站公開且 active 的文章。"

## Clarifications

### Session 2026-06-14

- Q: 搜尋的「包含字串匹配」應該如何定義實作期望？ → A: MVP 不要求專用搜尋索引或外部搜尋引擎，但必須維持任意位置包含匹配、排序、去重、分頁與 2 秒首批結果目標。
- Q: 使用者直接開啟 `/posts/search` 但 `q` 缺失、只有空白或 trim 後為空時，系統應該怎麼處理？ → A: 導回 `/posts`，不顯示搜尋結果頁。
- Q: 同一相關度層級內，如果多篇文章的發文時間完全相同，排序應如何穩定？ → A: 發文時間相同時，以文章 ID 新到舊作穩定 tie-breaker。

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 從文章河道搜尋公開文章 (Priority: P1)

使用者在 `/posts` 文章河道頂部輸入關鍵字後，可以按 Enter 或點擊搜尋按鈕送出；有效搜尋會導向獨立搜尋結果頁，讓使用者在不改變主河道狀態的情況下查看全站公開文章結果。

**Why this priority**: 這是功能的核心入口與主要價值；沒有可靠的搜尋送出與獨立結果頁，使用者無法完成找文章的主要任務。

**Independent Test**: 可透過在 `/posts` 輸入有效關鍵字並送出，驗證頁面導向 `/posts/search?q={keyword}`、搜尋框保留關鍵字，且結果不混入 `/posts` 主河道狀態。

**Acceptance Scenarios**:

1. **Given** 使用者位於 `/posts`，**When** 使用者輸入 trim 後有效的關鍵字並按 Enter，**Then** 系統導向 `/posts/search?q={keyword}` 並顯示符合條件的搜尋結果。
2. **Given** 使用者位於 `/posts`，**When** 使用者輸入 trim 後有效的關鍵字並點擊搜尋按鈕，**Then** 系統執行同樣的搜尋流程。
3. **Given** 使用者位於 `/posts` 且搜尋框內容 trim 後為空字串，**When** 使用者送出搜尋，**Then** 系統不執行搜尋、不離開目前頁面，並提示使用者輸入關鍵字。
4. **Given** 使用者位於 `/posts/search?q={keyword}`，**When** 使用者重新整理頁面，**Then** 頁面保留搜尋關鍵字並重新顯示對應結果。
5. **Given** 使用者位於搜尋結果頁，**When** 使用者清空搜尋框後送出，**Then** 系統導回 `/posts`。

---

### User Story 2 - 以可理解的結果找到命中文章 (Priority: P2)

使用者在搜尋結果頁可以看到符合關鍵字的公開 active 文章，並透過高亮與命中片段理解每筆結果為何出現；結果排序先呈現標題命中，再呈現僅內文命中，且同層級依發文時間新到舊排列。

**Why this priority**: 搜尋結果若不清楚或排序不符合期待，使用者仍需要逐篇打開文章判斷內容，會降低搜尋效率。

**Independent Test**: 可準備標題命中、內文命中、標題與內文皆命中、不同發文時間的公開文章，驗證結果收錄、排序、高亮與摘要片段。

**Acceptance Scenarios**:

1. **Given** 有公開 active 文章的標題包含關鍵字，**When** 使用者搜尋該關鍵字，**Then** 該文章出現在搜尋結果中。
2. **Given** 有公開 active 文章的內文包含關鍵字但標題不包含，**When** 使用者搜尋該關鍵字，**Then** 該文章出現在搜尋結果中。
3. **Given** 結果同時包含標題命中與僅內文命中文章，**When** 搜尋結果顯示，**Then** 標題命中文章排在僅內文命中文章之前。
4. **Given** 多篇文章位於同一相關度層級，**When** 搜尋結果顯示，**Then** 文章依發文時間新到舊排序；若發文時間相同，依文章 ID 新到舊穩定排序。
5. **Given** 一篇文章標題與內文都命中，**When** 搜尋結果排序，**Then** 該文章歸入標題命中層級。
6. **Given** 結果的命中位置在內文，**When** 結果卡片顯示摘要，**Then** 摘要顯示第一個命中片段附近內容，而不是固定顯示文章開頭。
7. **Given** 搜尋結果包含關鍵字，**When** 結果卡片顯示，**Then** 命中的關鍵字在標題或摘要中被高亮。

---

### User Story 3 - 在搜尋結果中延續文章河道互動 (Priority: P2)

使用者在搜尋結果頁看到的文章卡片應延續文章河道心智：顯示作者、時間、標題、內容摘要、按讚數、留言數與收藏狀態；登入使用者能看到自己的個人化狀態與作者操作，匿名使用者可搜尋與閱讀公開文章但不能執行需登入的互動。

**Why this priority**: 搜尋是文章河道的延伸，不應讓使用者在結果頁失去熟悉的閱讀與互動判斷。

**Independent Test**: 可分別以匿名使用者、一般登入使用者、文章作者身分搜尋同一組文章，驗證卡片資訊、個人化狀態、作者操作與詳文導覽。

**Acceptance Scenarios**:

1. **Given** 匿名使用者位於搜尋結果頁，**When** 看到公開文章結果，**Then** 使用者可以搜尋、閱讀結果並進入詳文頁，但不能執行需要登入的互動。
2. **Given** 登入使用者位於搜尋結果頁，**When** 看到自己曾按讚或收藏的文章，**Then** 卡片顯示對應的已按讚與已收藏狀態。
3. **Given** 登入使用者是某篇結果文章的作者，**When** 該文章卡片顯示，**Then** 使用者可看到自己的作者操作選單。
4. **Given** 使用者點擊搜尋結果的標題或留言數，**When** 導覽完成，**Then** 使用者進入對應文章詳文頁。

---

### User Story 4 - 處理大量結果、空結果與錯誤狀態 (Priority: P3)

當搜尋結果很多時，使用者可以透過無限捲動載入更多結果；初始搜尋、載入更多、無結果與搜尋失敗時都有清楚狀態，且錯誤狀態保留目前關鍵字並提供重試。

**Why this priority**: 這些狀態保證搜尋體驗完整且可恢復，避免使用者誤以為系統沒有回應或結果遺失。

**Independent Test**: 可用大量結果、零結果與失敗情境驗證載入狀態、重複結果排除、空狀態文案、錯誤區塊與重試行為。

**Acceptance Scenarios**:

1. **Given** 搜尋結果數量超過初始可顯示數量，**When** 使用者捲動到結果底部，**Then** 系統載入更多結果並顯示清楚的載入狀態。
2. **Given** 使用者載入更多搜尋結果，**When** 新結果加入頁面，**Then** 已顯示過的同一篇文章不會重複出現。
3. **Given** 搜尋沒有任何結果，**When** 搜尋完成，**Then** 頁面顯示 `找不到符合「{q}」的文章`，並提示使用者換個關鍵字。
4. **Given** 搜尋失敗，**When** 錯誤狀態顯示，**Then** 頁面保留搜尋框與目前關鍵字，顯示可見錯誤區塊與重試按鈕，例如「搜尋失敗，請稍後再試」。
5. **Given** 搜尋失敗後使用者點擊重試，**When** 重試執行，**Then** 系統使用目前關鍵字重新搜尋。

---

### Edge Cases

- 關鍵字前後有空白時，搜尋以 trim 後內容為準；trim 後為空字串不得執行搜尋。
- trim 後 1 個字元是有效關鍵字，必須可以搜尋。
- 英文字母搜尋大小寫不敏感；中文搜尋不要求斷詞、同義詞或模糊比對。
- 搜尋關鍵字只在文章標題與文章內文中比對；作者名稱、留言、編輯歷史、tag 與 category 不列入搜尋。
- 搜尋結果必須排除 soft-deleted 文章與 account-deletion-hidden 文章。
- 搜尋結果可見性必須與文章列表、詳文頁、會員頁、收藏頁一致，不能讓任何已刪除或隱藏文章重新出現。
- 搜尋結果頁不顯示發文入口；發文入口仍只屬於 `/posts` 文章河道。
- 使用者從搜尋頁返回 `/posts` 時，`/posts` 仍維持最新文章河道，不被搜尋結果取代。
- 搜尋結果載入更多時，若文章已在目前結果中出現，不能再次顯示同一篇文章。
- 使用者直接開啟 `/posts/search` 且 `q` 缺失、只有空白或 trim 後為空時，系統導回 `/posts`，不顯示搜尋結果頁。

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: `/posts` 文章河道頂部 MUST 顯示搜尋列，讓使用者輸入文章搜尋關鍵字。
- **FR-002**: 使用者 MUST be able to 透過 Enter 鍵或搜尋按鈕送出搜尋。
- **FR-003**: 系統 MUST 在送出前 trim 搜尋框內容；trim 後空字串不得執行搜尋，且 MUST 顯示輸入關鍵字提示。
- **FR-004**: 系統 MUST 在 `/posts` 成功送出有效關鍵字後，導向 `/posts/search?q={keyword}`。
- **FR-005**: `/posts/search` MUST 是獨立搜尋結果頁，不得把搜尋結果混入或取代 `/posts` 主河道狀態。
- **FR-006**: `/posts/search` 頁面上方 MUST 保留搜尋列，並顯示目前搜尋關鍵字，讓使用者可修改後再次搜尋。
- **FR-007**: 使用者 MUST be able to 重新整理 `/posts/search?q={keyword}` 後仍看到相同關鍵字與對應搜尋結果。
- **FR-008**: 使用者在搜尋結果頁清空關鍵字並送出時，系統 MUST 導回 `/posts`。
- **FR-009**: 搜尋範圍 MUST 包含全站所有公開且 active 的文章，不限於目前 `/posts` 河道已載入文章。
- **FR-010**: 搜尋匹配欄位 MUST 只包含文章標題與文章內文。
- **FR-011**: 搜尋 MUST 不匹配作者名稱、留言、編輯歷史、tag 或 category。
- **FR-012**: 搜尋 MUST 採包含字串匹配；標題或內文任一處包含關鍵字即為命中。
- **FR-013**: 英文字母搜尋 MUST 大小寫不敏感。
- **FR-014**: 中文搜尋 MUST 不要求斷詞、同義詞、模糊搜尋或語意擴展。
- **FR-015**: trim 後 1 個字元 MUST 被視為有效搜尋關鍵字。
- **FR-016**: 搜尋結果 MUST 排除 soft-deleted 文章與 account-deletion-hidden 文章。
- **FR-017**: 搜尋結果可見性 MUST 與文章列表、詳文頁、會員頁、收藏頁的公開文章可見性規則一致。
- **FR-018**: 搜尋結果排序 MUST 先顯示標題命中文章，再顯示僅內文命中文章。
- **FR-019**: 同一相關度層級內，搜尋結果 MUST 依發文時間由新到舊排序；發文時間相同時，MUST 依文章 ID 由新到舊穩定排序。
- **FR-020**: 同一篇文章標題與內文都命中時，MUST 視為標題命中層級。
- **FR-021**: 搜尋結果卡片 MUST 顯示作者、時間、標題、內容摘要、按讚數、留言數與收藏狀態。
- **FR-022**: 搜尋結果卡片 MUST 沿用文章河道卡片的核心互動，包括詳文導覽、按讚、收藏，以及作者可用的編輯與刪除入口。
- **FR-023**: 匿名使用者 MUST be able to 搜尋公開文章並進入詳文頁閱讀。
- **FR-024**: 匿名使用者 MUST NOT be able to 執行需要登入的互動。
- **FR-025**: 登入使用者 MUST 在搜尋結果中看到個人化狀態，包括已按讚、已收藏與自己的作者操作選單。
- **FR-026**: 使用者點擊搜尋結果的標題或留言數時，系統 MUST 導向對應文章詳文頁。
- **FR-027**: 搜尋結果 MUST 高亮關鍵字，讓使用者辨識命中位置。
- **FR-028**: 內文命中時，結果摘要 MUST 顯示第一個命中片段附近內容，而不是固定顯示文章開頭。
- **FR-029**: 搜尋結果很多時，系統 MUST 支援無限捲動載入更多結果。
- **FR-030**: 初始搜尋與載入更多時，系統 MUST 顯示清楚載入狀態，並沿用文章頁既有骨架屏體驗。
- **FR-031**: 載入更多搜尋結果時，系統 MUST 防止同一篇文章在同一組搜尋結果中重複顯示。
- **FR-032**: 無結果時，系統 MUST 顯示 `找不到符合「{q}」的文章`。
- **FR-033**: 無結果狀態 MUST 顯示副文案，提示使用者換個關鍵字。
- **FR-034**: 搜尋失敗時，系統 MUST 顯示可見錯誤區塊與重試按鈕，文案可為「搜尋失敗，請稍後再試」或等價清楚訊息。
- **FR-035**: 搜尋失敗狀態 MUST 保留搜尋框與目前關鍵字。
- **FR-036**: 使用者點擊重試時，系統 MUST 使用目前關鍵字重新搜尋。
- **FR-037**: `/posts` MUST 維持最新文章河道，不因搜尋功能改變排序、載入狀態或已載入內容。
- **FR-038**: `/posts` 的發文入口、發文與編輯草稿、按讚、收藏、作者編輯與刪除、編輯歷史、詳文連結、未登入閱讀等既有行為 MUST 維持不退化。
- **FR-039**: 搜尋功能 MUST NOT 改變文章建立、編輯、刪除、留言、按讚、收藏的既有權限與互動規則。
- **FR-040**: 搜尋結果頁 MUST NOT 顯示發文入口；使用者要發文時仍回到 `/posts` 文章河道。
- **FR-041**: 使用者直接開啟 `/posts/search` 且 `q` 缺失、只有空白或 trim 後為空時，系統 MUST 導回 `/posts`，且 MUST NOT 顯示搜尋結果頁。

### Key Entities

- **搜尋關鍵字**: 使用者輸入並送出的文字；以 trim 後內容作為搜尋依據，可從搜尋結果頁網址還原。
- **公開 active 文章**: 可被一般使用者閱讀、未被刪除、未因帳號刪除而隱藏的文章；包含作者、時間、標題、內文摘要、互動統計與公開可見狀態。
- **搜尋結果**: 符合搜尋關鍵字與可見性規則的文章集合；每筆結果保留文章河道卡片的核心資訊與互動。
- **相關度層級**: 搜尋結果排序用的產品概念；標題命中優先於僅內文命中。
- **文章 ID**: 用於同一相關度層級且發文時間相同時的穩定排序 tie-breaker。
- **個人化互動狀態**: 登入使用者在文章上的已按讚、已收藏與作者操作可見性；匿名使用者不具備需登入互動。

### Assumptions

- 搜尋結果頁使用與文章河道一致的公開文章可見性判斷，作為排除已刪除或隱藏文章的依據。
- 關鍵字高亮只需要呈現使用者實際搜尋的 trim 後關鍵字，不需要額外高亮同義詞、斷詞結果或模糊命中。
- 「命中片段附近內容」表示摘要足以讓使用者看見第一個內文命中的上下文；具體長度可由後續設計決定，但不得退回固定文章開頭。
- 搜尋結果頁的卡片互動應以既有文章河道卡片能力為準；若匿名使用者觸發需登入互動，維持既有登入要求體驗。
- MVP 不要求專用搜尋索引或外部搜尋引擎；無論採用何種候選集合策略，搜尋結果都不得降低任意位置包含匹配、排序、去重、無限捲動與 2 秒首批結果目標。

### Out of Scope

- 作者搜尋。
- 留言搜尋。
- tag 或 category 搜尋。
- 熱門搜尋、搜尋歷史、推薦關鍵字。
- 排序切換 UI。
- 全文搜尋引擎、模糊搜尋、同義詞搜尋或中文斷詞規格。
- 改造文章詳文頁或發文流程，除非是維持搜尋結果導覽一致性所必需。

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 在可用性驗收中，至少 90% 測試使用者能從 `/posts` 以有效關鍵字完成搜尋並在 30 秒內抵達搜尋結果頁。
- **SC-002**: 100% 驗收情境中，有效關鍵字送出後會導向 `/posts/search?q={keyword}`，且重新整理後保留關鍵字與結果。
- **SC-003**: 100% 搜尋結果驗收資料中，標題命中文章排序優先於僅內文命中文章；同層級文章依發文時間新到舊排列，且同一發文時間時依文章 ID 新到舊穩定排列。
- **SC-004**: 100% 搜尋結果驗收資料中，soft-deleted 與 account-deletion-hidden 文章不會出現在結果。
- **SC-005**: 95% 一般搜尋在正常連線下，使用者可在 2 秒內看見第一批結果、空狀態或錯誤狀態。
- **SC-006**: 100% 內文命中結果會在摘要中呈現第一個命中片段附近內容，並高亮關鍵字。
- **SC-007**: 100% 既有 `/posts` 文章河道驗收情境維持通過，包括最新排序、無限捲動、發文入口、卡片互動與未登入閱讀。
- **SC-008**: 在使用者驗收中，至少 80% 測試使用者能正確說明一筆內文命中結果為何出現。
