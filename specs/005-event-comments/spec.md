# Feature Specification: 活動留言功能

**Feature Branch**: `005-event-comments`
**Created**: 2026-04-02
**Status**: Draft
**Input**: 活動詳情頁留言功能 — 使用者可在路線圖下方瀏覽、發表、編輯、刪除留言

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 瀏覽留言 (Priority: P1)

使用者進入活動詳情頁，在路線地圖下方看到留言區。留言以時間倒序排列（最新在最上方）。當留言數量多時，往下捲動會自動載入更多歷史留言。

**Why this priority**: 跑友在報名前或報名後，第一步就是想看其他人針對活動的討論——集合地點、路況、裝備等。瀏覽留言是跑友獲取活動資訊和了解參加氛圍的最基本途徑。

**Independent Test**: 進入任一活動詳情頁，在路線圖下方看到留言列表，可正常捲動瀏覽。

**Acceptance Scenarios**:

1. **Given** 活動有留言，**When** 使用者進入活動詳情頁，**Then** 路線圖下方顯示留言列表，最新留言在最上方
2. **Given** 活動有超過一頁的留言，**When** 使用者往下捲動到接近底部，**Then** 系統自動載入更多留言並附加到列表底部
3. **Given** 活動沒有留言，**When** 使用者進入活動詳情頁，**Then** 留言區顯示空狀態提示
4. **Given** 使用者未登入，**When** 使用者瀏覽留言區，**Then** 可以看到所有留言，但看不到發表留言的輸入框

---

### User Story 2 - 發表留言 (Priority: P1)

登入使用者在留言區底部看到一個浮動的輸入框和「送出」按鈕。輸入留言後點擊「送出」，系統顯示 loading 動畫，直到留言成功儲存。若儲存失敗，顯示「送出失敗，請再試一次」通知。

**Why this priority**: 跑友之間需要一個管道針對活動進行聯繫和溝通——詢問集合細節、分享路線心得、協調共乘等。發表留言讓這些交流得以發生，與瀏覽一起構成留言功能的 MVP。

**Independent Test**: 登入後在活動詳情頁留言區輸入內容並送出，確認留言出現在列表中。

**Acceptance Scenarios**:

1. **Given** 使用者已登入，**When** 在輸入框輸入文字並點擊「送出」，**Then** 按鈕顯示 loading 動畫，成功後新留言出現在列表最上方，輸入框清空，且該則新留言以背景色短暫閃爍的高亮效果吸引使用者注意
2. **Given** 使用者已登入，**When** 輸入框為空或只有空白，**Then** 「送出」按鈕顯示為不可點擊狀態
3. **Given** 使用者已登入且送出留言，**When** 儲存失敗，**Then** 顯示「送出失敗，請再試一次」通知，輸入框內容保留不清空
4. **Given** 浮動輸入框可見，**When** 留言列表捲動到最底部，**Then** 浮動輸入框不會遮擋最後一則留言

---

### User Story 3 - 編輯留言與編輯記錄 (Priority: P2)

留言者在自己的留言卡片右上角看到三點選單按鈕。點擊後展開選單，選擇「編輯留言」會彈出編輯視窗，輸入框預填原本留言內容。若內容沒有實質變更（去除前後空白後與原文相同），「完成編輯」按鈕為不可點擊狀態。編輯完成後，該留言會顯示「已編輯」標記。任何使用者點擊「已編輯」可查看該留言的完整編輯記錄。

**Why this priority**: 跑友可能需要修正留言中的錯誤資訊（如時間、地點寫錯）或補充細節，避免其他跑友收到錯誤資訊。編輯記錄讓其他跑友能追溯留言的修改歷程，增加資訊透明度。

**Independent Test**: 登入後對自己的留言執行編輯，確認更新成功、顯示「已編輯」標記，並可透過點擊「已編輯」查看編輯記錄。

**Acceptance Scenarios**:

1. **Given** 使用者是某則留言的作者，**When** 點擊該留言的三點選單 → 「編輯留言」，**Then** 彈出編輯視窗，輸入框自動填入原本的留言內容
2. **Given** 編輯視窗已開啟，**When** 輸入框內容 trim 後與原文相同，**Then** 「完成編輯」按鈕顯示為不可點擊狀態
3. **Given** 編輯視窗已開啟，**When** 輸入框內容 trim 後與原文不同並點擊「完成編輯」，**Then** 系統保留舊版本作為歷史記錄、更新留言內容並關閉編輯視窗
4. **Given** 編輯視窗已開啟，**When** 點擊「取消編輯」，**Then** 關閉編輯視窗，留言內容不變
5. **Given** 使用者不是某則留言的作者，**When** 查看該留言，**Then** 看不到三點選單按鈕
6. **Given** 一則留言曾被編輯過，**When** 任何使用者查看該留言，**Then** 時間旁顯示「已編輯」標記
7. **Given** 一則留言顯示「已編輯」標記，**When** 使用者點擊「已編輯」，**Then** 彈出「編輯記錄」視窗，依時間順序列出所有版本含當前版本（每版本包含內容和時間）

---

### User Story 4 - 刪除留言 (Priority: P2)

留言者透過三點選單選擇「刪除留言」，系統彈出自定義確認視窗（非瀏覽器原生 confirm）詢問「確定刪除留言？」。確認刪除後顯示 loading 動畫直到刪除完成。刪除失敗則顯示「刪除失敗，請再試一次」。

**Why this priority**: 活動資訊可能隨時變動，跑友需要能刪除已過時或不再正確的留言，避免誤導其他參與者。

**Independent Test**: 登入後對自己的留言執行刪除，確認留言從列表中移除。

**Acceptance Scenarios**:

1. **Given** 使用者是某則留言的作者，**When** 點擊三點選單 → 「刪除留言」，**Then** 彈出自定義確認視窗，顯示「確定刪除留言？」
2. **Given** 確認視窗已開啟，**When** 點擊「確定刪除」，**Then** 按鈕顯示 loading 動畫，刪除成功後留言從列表消失並關閉視窗
3. **Given** 確認視窗已開啟，**When** 點擊「確定刪除」但刪除失敗，**Then** 顯示「刪除失敗，請再試一次」通知
4. **Given** 確認視窗已開啟，**When** 點擊「取消刪除」，**Then** 關閉確認視窗，留言不受影響

---

### Edge Cases

- 使用者在留言送出過程中離開頁面或失去網路連線
- 留言者在編輯/刪除留言時，該留言已被後台刪除
- 多位使用者同時對同一則留言操作（一人編輯、一人刪除）
- 留言接近 500 字上限時的字數提示行為
- 無限捲動同時有新留言進來時的列表更新
- 浮動輸入框在不同螢幕尺寸下的行為

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 系統 MUST 在活動詳情頁路線圖下方顯示留言區
- **FR-002**: 系統 MUST 以時間倒序（新→舊）排列留言
- **FR-003**: 系統 MUST 在使用者捲動接近留言列表底部時自動載入更多留言（無限捲動）
- **FR-004**: 系統 MUST 對登入使用者在留言區底部顯示浮動輸入框和「送出」按鈕
- **FR-005**: 浮動輸入框 MUST NOT 遮擋最後一則留言的內容
- **FR-006**: 輸入框為空或只有空白時，「送出」按鈕 MUST 顯示為不可點擊狀態
- **FR-007**: 留言字數上限為 500 字，超過時 MUST 禁止送出並顯示字數超過提示
- **FR-008**: 送出留言時系統 MUST 顯示 loading 動畫，直到操作完成
- **FR-009**: 送出失敗時系統 MUST 顯示「送出失敗，請再試一次」通知，並保留輸入框內容
- **FR-010**: 送出成功後，新留言 MUST 以背景色短暫閃爍的高亮動畫效果標示，讓使用者一目瞭然自己剛發表的留言位置
- **FR-011**: 每則留言 MUST 顯示留言者頭像、名稱、發表時間和留言內容
- **FR-012**: 發表時間 MUST 以「月/日 時:分」格式顯示，採用 24 小時制（例如「4/2 14:30」）
- **FR-013**: 桌面端滑鼠 hover 發表時間時，MUST 顯示包含西元年份的完整時間 tooltip（例如「2026年4月2日 14:30」）；手機端點擊一下即可看到相同的完整時間
- **FR-014**: 已被編輯過的留言 MUST 在時間旁顯示「已編輯」標記
- **FR-015**: 點擊「已編輯」標記 MUST 開啟「編輯記錄」視窗，顯示該留言的所有版本（含當前版本），每個版本包含留言內容和該版本的時間
- **FR-016**: 編輯留言時系統 MUST 保留編輯前的內容作為歷史記錄，不得覆蓋
- **FR-017**: 留言作者 MUST 在自己的留言卡片右上角看到三點選單按鈕
- **FR-018**: 三點選單 MUST 包含「編輯留言」和「刪除留言」兩個選項
- **FR-019**: 「編輯留言」MUST 開啟編輯視窗，輸入框預填原本留言內容
- **FR-020**: 編輯視窗 MUST 包含「取消編輯」和「完成編輯」按鈕
- **FR-021**: 編輯內容 trim 後與原文相同時，「完成編輯」MUST 顯示為不可點擊狀態
- **FR-022**: 「刪除留言」MUST 開啟自定義確認視窗（非瀏覽器原生 confirm），顯示「確定刪除留言？」
- **FR-023**: 確認刪除後系統 MUST 顯示 loading 動畫直到刪除操作完成
- **FR-024**: 刪除失敗時系統 MUST 顯示「刪除失敗，請再試一次」通知
- **FR-025**: 未登入使用者 MUST 能瀏覽留言但 MUST NOT 看到發表留言的輸入框
- **FR-026**: 非留言作者 MUST NOT 看到該留言的三點選單按鈕

### Key Entities

- **Comment（留言）**: 留言的核心資料，包含作者身份（ID、名稱、頭像 URL）、留言文字內容、建立時間、是否已編輯。每則留言隸屬於一個活動。
- **CommentHistory（編輯記錄）**: 每次編輯前的留言內容快照，包含編輯前的文字內容和該版本的時間。每則留言可有零到多筆編輯記錄。
- **Event（活動）**: 既有實體，留言的父層容器。每個活動可擁有多則留言。

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 使用者進入活動詳情頁後，留言列表在 3 秒內完成首次載入
- **SC-002**: 送出留言後，新留言在 3 秒內出現在列表中
- **SC-003**: 捲動觸發載入更多留言時，新留言在 2 秒內附加到列表中
- **SC-004**: 所有操作（送出/編輯/刪除）100% 提供 loading 狀態回饋和失敗通知
- **SC-005**: 留言作者對自己留言的編輯和刪除操作，90% 在首次嘗試即成功完成

## Assumptions

- 每則留言為純文字，不支援圖片、表情符號或富文本格式
- 每次無限捲動載入一批留言（合理數量，如 10-20 則）
- 留言無巢狀回覆功能（扁平列表）
- 留言無按讚或其他互動按鈕
- 留言的顯示時間格式為「月/日 時:分」24 小時制（如「4/2 14:30」），hover/tap 顯示含年份完整時間
- 留言字數上限 500 字
- 活動建立者無特殊的留言管理權限（不能刪除他人留言）

## Clarifications

### Session 2026-04-02

- Q: 留言是否有字數上限？ → A: 500 字上限，超過禁止送出並顯示提示
- Q: 「編輯記錄」視窗是否包含當前版本？ → A: 列出全部版本（原始 + 所有編輯 + 當前版本）
