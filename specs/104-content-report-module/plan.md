## Summary

Phase 1 builds a reusable server-side content report ingestion module. All report creation goes through `POST /api/reports`; the server authenticates the bearer token with Firebase Admin, validates the request, resolves the target with server-side Firestore reads, writes a deterministic report document to `reports`, and returns stable status/message contracts. Client code never directly reads or writes `reports`.

The API/schema supports `post`, `postComment`, `event`, and `eventComment` from the start. Phase 1 UI only wires `post` and `postComment` entry points on `/posts`, `/posts/search`, and `/posts/[id]`, including normal comments and notification `?commentId=` target comments. Event and event-comment UI entry points remain explicitly out of scope until Phase 2.

## Technical Context

**Language/Version**: JavaScript ES6+ with JSDoc and `checkJs: true`; Next.js 15 App Router; React 19.
**Primary Dependencies**: Firebase v9+ client facades already in the app, Firebase Admin server helpers from `src/config/server/firebase-admin-app.js`, Vitest, React Testing Library, `@firebase/rules-unit-testing`, Playwright for browser/E2E verification.
**Storage**: Firestore `reports` top-level collection, written only by server/Admin code. Existing content remains in `posts`, `posts/{postId}/comments`, `events`, and `events/{eventId}/comments`.
**Testing**: Vitest unit/integration/server tests, Firestore rules emulator tests, and Playwright/browser visual verification. Planned tests live under `tests/**`, never under `specs/**`.
**Target Platform**: Next.js web app with serverless App Router API route and browser UI.
**Project Type**: Full-stack web application inside a single Next.js repo.
**Performance Goals**: One report submit performs bounded server work: authenticate token, validate payload, resolve exactly one target plus parent visibility for comment targets, and attempt one deterministic report create. No Phase 1 report list queries, reporter-history queries, notification fan-out, media snapshot fetches, or composite indexes.
**Constraints**: Preserve forward-only dependencies `Types -> Config -> Repo -> Service -> Runtime -> UI`; keep App Router route thin; Firebase SDK/Admin access stays in `src/config/**` and `src/repo/**`; `src/service/**` owns validation/normalization; `src/runtime/**` owns use-case orchestration and hooks; UI is render/interaction only. `details` is trimmed and capped at 500 characters; `other` requires non-empty details; `sourcePath` is sanitized to same-app relative paths with max 1024 characters, and missing/invalid/over-limit values fall back to the server-generated targetPath without truncation. Target snapshot is generated by server and excludes media URLs, full author profiles, reporter display names, and reporter profiles.
**Scale/Scope**: One API route, one reusable report domain module, one server target resolver, one server repo, one Firestore rules deny block, reusable client runtime/dialog pieces, and Phase 1 post/post-comment UI wiring only.

## Complexity Tracking

No constitution violations are required for this plan.

### Gate Notes

- **Deterministic id vs transaction**: duplicate prevention is modeled as an atomic create to `reports/{sha256(reporterUid + targetType + targetKey)}`. This avoids report list queries and avoids overwriting existing reports. Future aggregate counters or moderation workflows are out of scope and would require separate transaction planning.
- **Manual AGENTS marker patch**: use a marker-only patch for `AGENTS.md`; do not run `.specify/scripts/bash/update-agent-context.sh`.
- **No Phase 1 indexes**: because Phase 1 has no admin list, reporter history, or report lookup UI, it must not add Firestore composite indexes.
- **No client trust expansion**: client preview and sourcePath are contextual only; server target reads and snapshots are the authority.
