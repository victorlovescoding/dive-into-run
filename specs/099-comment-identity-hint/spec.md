# Feature Specification: 留言輸入身份提示一致化

**Feature Branch**: `099-comment-identity-hint`
**Created**: 2026-06-06
**Status**: Draft
**Input**: User description: "PRD: 留言輸入身份提示一致化"

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 已登入使用者確認留言身份 (Priority: P1)

已登入使用者在文章頁或活動頁準備留言時，可以在底部留言輸入區左側看到自己的圓形大頭貼，並在輸入前確認目前留言帳號。

**Why this priority**: 這是本需求的主要價值；它直接降低多帳號、換帳號或登入狀態不明確時用錯帳號留言的風險。

**Independent Test**: 以有大頭貼的已登入帳號分別開啟文章頁與活動頁，確認兩個頁面的底部留言輸入區都顯示同一個目前帳號的大頭貼，且可正常輸入留言。

**Acceptance Scenarios**:

1. **Given** 使用者已登入且帳號有大頭貼，**When** 使用者開啟文章頁，**Then** 頁面底部顯示留言輸入區，左側顯示該使用者的大頭貼。
2. **Given** 使用者已登入且帳號有大頭貼，**When** 使用者開啟活動頁，**Then** 頁面底部顯示留言輸入區，左側顯示該使用者的大頭貼。
3. **Given** 使用者已登入，**When** 使用者查看文章頁與活動頁留言輸入區，**Then** 兩者的大頭貼位置、輸入欄、送出按鈕與底部固定呈現方式一致。

---

### User Story 2 - 沒有大頭貼時仍可辨識留言入口 (Priority: P2)

已登入但未設定大頭貼的使用者在文章頁或活動頁留言前，仍會看到預設大頭貼作為身份提示，不會看到缺圖或空白佔位。

**Why this priority**: 身份提示必須對所有已登入使用者成立，否則未設定大頭貼的帳號會回到不明確狀態。

**Independent Test**: 以沒有大頭貼的已登入帳號分別開啟文章頁與活動頁，確認留言輸入區左側顯示預設大頭貼，且不影響輸入與送出。

**Acceptance Scenarios**:

1. **Given** 使用者已登入但沒有大頭貼，**When** 使用者開啟文章頁，**Then** 留言輸入區左側顯示預設大頭貼。
2. **Given** 使用者已登入但沒有大頭貼，**When** 使用者開啟活動頁，**Then** 留言輸入區左側顯示預設大頭貼。

---

### User Story 3 - 未登入使用者不看到可留言欄位 (Priority: P2)

未登入使用者開啟文章頁或活動頁時，不會看到底部留言輸入區，避免誤以為目前可以直接留言。

**Why this priority**: 這防止未登入狀態造成誤解，也讓文章頁與活動頁的權限呈現一致。

**Independent Test**: 在未登入狀態分別開啟文章頁與活動頁，確認底部沒有留言輸入欄、大頭貼或送出按鈕。

**Acceptance Scenarios**:

1. **Given** 使用者未登入，**When** 使用者開啟文章頁，**Then** 不顯示留言輸入區。
2. **Given** 使用者未登入，**When** 使用者開啟活動頁，**Then** 不顯示留言輸入區。
3. **Given** 使用者未登入，**When** 使用者查看文章頁或活動頁底部，**Then** 不新增登入提示或登入導流。

---

### User Story 4 - 留言送出狀態一致且避免誤送 (Priority: P3)

已登入使用者在文章頁或活動頁輸入留言時，只有有效留言內容可以送出；送出成功後輸入欄會清空，送出中不能重複送出。

**Why this priority**: 身份提示之外，兩個頁面的留言輸入互動也必須一致，避免其中一個頁面留下可重複送出或無效送出的差異。

**Independent Test**: 在文章頁與活動頁分別測試空白留言、超過既有限制的留言、有效留言與送出中狀態，確認按鈕狀態與送出結果一致。

**Acceptance Scenarios**:

1. **Given** 使用者已登入且留言內容為空白或只有空白字元，**When** 使用者查看留言輸入區，**Then** 不能送出留言。
2. **Given** 使用者已登入且留言內容超過既有長度限制，**When** 使用者查看留言輸入區，**Then** 不能送出留言。
3. **Given** 使用者已登入且留言內容有效，**When** 使用者送出留言且送出成功，**Then** 輸入欄清空。
4. **Given** 使用者已登入且留言正在送出中，**When** 使用者再次嘗試送出，**Then** 系統避免重複送出。

### Edge Cases

- 使用者資料沒有大頭貼或大頭貼無法顯示時，留言輸入區仍顯示預設大頭貼。
- 使用者從已登入切換為未登入後，文章頁與活動頁都不再顯示留言輸入區。
- 使用者從未登入切換為已登入後，文章頁與活動頁都顯示包含目前帳號大頭貼的留言輸入區。
- 留言輸入區固定在畫面底部時，手機與桌機畫面中的輸入欄、按鈕、大頭貼不得重疊或截斷。
- 大頭貼只作為身份提示，不可點擊，也不導向個人頁。

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 系統 MUST 在已登入使用者的文章頁底部顯示留言輸入區。
- **FR-002**: 系統 MUST 在已登入使用者的活動頁底部顯示留言輸入區。
- **FR-003**: 文章頁與活動頁的留言輸入區 MUST 使用一致的視覺結構：左側圓形使用者大頭貼、中間留言輸入欄、右側送出按鈕。
- **FR-004**: 留言輸入區 MUST 固定顯示於畫面底部，並在手機與桌機尺寸下維持可辨識、可操作、不重疊、不截斷。
- **FR-005**: 系統 MUST 在留言輸入區左側顯示目前登入使用者的大頭貼。
- **FR-006**: 若目前登入使用者沒有可用大頭貼，系統 MUST 顯示預設大頭貼。
- **FR-007**: 大頭貼 MUST 僅作為身份提示，不可點擊，且不得導向個人頁。
- **FR-008**: 未登入使用者在文章頁與活動頁 MUST 都看不到留言輸入區。
- **FR-009**: 未登入狀態 MUST NOT 新增登入提示區塊或登入導流。
- **FR-010**: 使用者 MUST 能在已登入狀態透過底部留言輸入區輸入留言文字。
- **FR-011**: 留言內容為空白或只有空白字元時，系統 MUST 阻止送出。
- **FR-012**: 留言內容超過既有長度限制時，系統 MUST 阻止送出。
- **FR-013**: 留言內容有效時，使用者 MUST 能送出留言。
- **FR-014**: 留言送出成功後，系統 MUST 清空輸入欄。
- **FR-015**: 留言送出中，系統 MUST 避免同一則留言被重複送出。
- **FR-016**: 本次變更 MUST NOT 改變留言列表樣式、留言排序、通知、編輯、刪除邏輯。
- **FR-017**: 本次變更 MUST 沿用既有錯誤處理，不新增新的錯誤提示設計。

### Key Entities

- **目前使用者**: 代表目前登入狀態與留言身份；包含是否已登入、可顯示的大頭貼或預設大頭貼狀態。
- **留言輸入區**: 代表文章頁與活動頁底部的留言輸入介面；包含身份提示、文字輸入欄、送出按鈕與送出狀態。
- **留言草稿**: 代表使用者尚未送出的文字內容；需要符合非空白與既有長度限制。

### Assumptions

- 文章頁既有留言輸入區已具備目前帳號大頭貼提示；本需求要求活動頁與文章頁一致，並保留文章頁既有可用行為。
- 留言長度限制沿用目前產品既有規則，本需求不新增或修改長度數值。
- 預設大頭貼沿用產品既有預設頭像資源或既有缺圖替代樣式。
- 本需求只處理留言輸入區，不處理留言列表、排序、通知、編輯、刪除或登入導流。

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 在文章頁與活動頁各 100% 的已登入驗收案例中，使用者都能在底部留言輸入區左側看到目前帳號的大頭貼或預設大頭貼。
- **SC-002**: 在文章頁與活動頁各 100% 的未登入驗收案例中，頁面都不顯示留言輸入區。
- **SC-003**: 在文章頁與活動頁各 100% 的留言有效性驗收案例中，空白留言與超過既有長度限制的留言都不可送出，有效留言可送出。
- **SC-004**: 在送出成功的驗收案例中，文章頁與活動頁的留言輸入欄 100% 會清空。
- **SC-005**: 在送出中狀態的驗收案例中，文章頁與活動頁都能避免同一則留言被重複送出。
- **SC-006**: 在手機與桌機驗收尺寸中，文章頁與活動頁的留言輸入區都沒有大頭貼、輸入欄或送出按鈕重疊、截斷或不可操作的情況。
- **SC-007**: 在可用性檢查中，已登入使用者能在 2 秒內辨識目前留言帳號的大頭貼提示。
