# Feature Specification: 留言通知擴充 (Comment Notifications)

**Feature Branch**: `015-comment-notifications`
**Created**: 2026-04-14
**Status**: Draft
**Input**: User description: "我想在現有的通知系統上擴充三個功能- 我留言過的文章有新留言 → 收到通知、我留言過的活動有新留言 → 收到通知、我參加的活動有新留言 → 收到通知"

## Clarifications

### Session 2026-04-14

- Q: 主揪人是否「永遠」收到活動留言通知（不需留言或報名參加就自動觸發），還是只有當主揪人「剛好也是留言者或參加者」時才改用主揪人身份的訊息？ → A: 永遠觸發。主揪人身份自動觸發通知，類似文章作者收到 `post_new_comment` 的邏輯。
- Q: 主揪人同時是留言者或參加者時，通知如何去重？ → A: 主揪人身份優先於所有其他身份，只收到一則「你主辦的活動…」通知。
- Q: 參加者 + 留言者的去重優先順序？ → A: 參加者身份優先。訊息為「你參加的活動…」，因為「要去參加」的連結比「留過言」更強。
- Q: 主揪人是否會出現在活動的參加者名單中？ → A: 不會。系統設計上主揪人建立活動即代表會參加，不在 participants 名單中。故「主揪人 + 參加者」的身份組合在系統中不存在。
- Q: 使用者取消參加活動後，該活動有新留言時通知行為為何？ → A: 以留言當下的參加者名單為準。取消參加後不再收到「你參加的活動…」通知，但若該使用者曾留言，仍透過「留言跟帖」身份收到「你留言過的活動…」通知。此為去重邏輯查詢當前狀態的自然結果，無需額外處理。
- Q: 活動已結束（日期已過）後有人留言，是否仍發送通知？ → A: 是，不區分活動是否結束。活動後留言（心得、照片、感謝主揪）對社群黏性重要，且通知邏輯不需額外判斷活動狀態。
- Q: 所有留言通知（文章 + 活動）點擊後是否都需要 scroll-to-comment + highlight？ → A: 是。所有留言通知點擊後都必須導航至該頁面、滾動至該則留言並套用高光特效，沿用現有 `post_new_comment` 的 highlight 動畫。活動頁面需新增此功能。

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 文章留言跟帖通知 (Priority: P1)

使用者在某篇文章留過言之後，當其他人在同一篇文章新增留言時，該使用者收到通知。通知訊息為「你留言過的文章『{文章標題}』有一則新的留言」。點擊通知後導航至該文章頁面並滾動至該則新留言處（沿用現有 `post_new_comment` 的導航行為）。

**Why this priority**: 文章留言是目前最活躍的互動功能，現有系統僅通知文章作者，其他留言者無法得知後續討論，導致對話容易斷裂。擴充此通知能直接提升使用者回訪與互動率。

**Independent Test**: 建立一篇文章，User A 留言，User B 留言後驗證 User A 收到通知，且點擊通知可導航至該文章留言處。

**Acceptance Scenarios**:

1. **Given** User A 在 Post X 留過言，**When** User B 在 Post X 新增留言，**Then** User A 收到通知「你留言過的文章『{Post X 標題}』有一則新的留言」
2. **Given** User A、User C 都在 Post X 留過言，**When** User B 在 Post X 新增留言，**Then** User A 和 User C 各收到一則通知
3. **Given** User A 在 Post X 留過言，**When** User A 自己再次在 Post X 留言，**Then** User A 不會收到通知（不通知自己）
4. **Given** User A 是 Post X 的作者且也在自己文章留過言，**When** User B 留言，**Then** User A 只收到一則通知（既有的「你的文章有新留言」），不額外收到跟帖通知
5. **Given** User A 在 Post X 留過多則留言，**When** User B 新增一則留言，**Then** User A 只收到一則通知（不因留言數量重複）
6. **Given** User A 收到文章跟帖通知，**When** 點擊該通知，**Then** 導航至該文章頁面、滾動至該則留言並套用高光特效（沿用既有 highlight 動畫）

---

### User Story 2 - 活動參加者留言通知 (Priority: P1)

使用者已報名參加某活動，當有人在該活動新增留言時，該使用者收到通知。通知訊息為「你參加的活動『{活動標題}』有一則新的留言」。點擊通知後導航至活動詳文頁、滾動至該則留言並套用高光特效。

**Why this priority**: 參加者已明確表達對活動的興趣，活動留言往往涉及集合時間、地點變更等重要資訊，參加者需要即時得知。

**Independent Test**: 建立一個活動，User A 報名參加，User B 留言後驗證 User A 收到通知。

**Acceptance Scenarios**:

1. **Given** User A 已參加 Event X，**When** User B 在 Event X 新增留言，**Then** User A 收到通知「你參加的活動『{Event X 標題}』有一則新的留言」
2. **Given** User A、User C 都已參加 Event X，**When** User B 在 Event X 新增留言，**Then** User A 和 User C 各收到一則通知
3. **Given** User A 已參加 Event X，**When** User A 自己在 Event X 留言，**Then** User A 不會收到通知
4. **Given** Event X 有 20 位參加者，**When** 某位非參加者 User B 留言，**Then** 20 位參加者各收到一則通知
5. **Given** User A 收到活動參加者留言通知，**When** 點擊該通知，**Then** 導航至 Event X 詳文頁、滾動至該則留言並套用高光特效

---

### User Story 3 - 活動留言跟帖通知 (Priority: P1)

使用者在某活動留過言之後，當其他人在同一活動新增留言時，該使用者收到通知。通知訊息為「你留言過的活動『{活動標題}』有一則新的留言」。點擊通知後導航至活動詳文頁、滾動至該則留言並套用高光特效。若使用者同時是活動參加者且曾留言，只收到一則通知（去重），優先使用「參加者」的通知訊息（「你參加的活動…」）。此通知主要涵蓋非參加者的留言者（例如路過提問的使用者）。

**Why this priority**: 與文章跟帖通知邏輯一致。在活動留言區討論過的使用者需要追蹤後續對話。

**Independent Test**: User A（非參加者）在 Event X 留言，User B 留言後驗證 User A 收到通知。另測試 User A 同時是參加者兼留言者時只收到一則「你參加的活動…」通知。

**Acceptance Scenarios**:

1. **Given** User A 在 Event X 留過言（但未參加），**When** User B 在 Event X 新增留言，**Then** User A 收到通知「你留言過的活動『{Event X 標題}』有一則新的留言」
2. **Given** User A 在 Event X 留過言，**When** User A 自己再次在 Event X 留言，**Then** User A 不會收到通知
3. **Given** User A 同時是 Event X 的參加者且留過言，**When** User B 留言，**Then** User A 只收到一則通知（去重），訊息為「你參加的活動…」（參加者身份優先）
4. **Given** User A 收到活動跟帖通知，**When** 點擊該通知，**Then** 導航至 Event X 詳文頁、滾動至該則留言並套用高光特效

---

### User Story 4 - 活動主揪人留言通知 (Priority: P1)

使用者是某活動的主揪人（建立者），當有人在該活動新增留言時，主揪人收到通知。通知訊息為「你主辦的活動『{活動標題}』有一則新的留言」。主揪人不需報名參加或留言即自動觸發此通知，類似文章作者收到「你的文章有新留言」的邏輯。若主揪人同時也是留言者，只收到一則「你主辦的…」通知（主揪人身份優先級最高）。注意：系統設計上主揪人不在 participants 名單中，故「主揪人 + 參加者」的組合不存在。

**Why this priority**: 主揪人是活動的建立者與負責人，活動留言可能涉及參加者的問題、集合地點確認等，主揪人理應第一時間得知。與文章作者通知對稱——你創建的內容有互動，你就應該知道。

**Independent Test**: 建立一個活動，User B 留言後驗證主揪人收到通知。另測試主揪人同時是留言者時只收到一則通知。

**Acceptance Scenarios**:

1. **Given** User A 是 Event X 的主揪人（未留言），**When** User B 在 Event X 新增留言，**Then** User A 收到通知「你主辦的活動『{Event X 標題}』有一則新的留言」
2. **Given** User A 是 Event X 的主揪人且也在 Event X 留過言，**When** User B 留言，**Then** User A 只收到一則通知「你主辦的活動…」（不額外收到跟帖通知）
3. **Given** User A 是 Event X 的主揪人，**When** User A 自己在 Event X 留言，**Then** User A 不會收到通知（不通知自己）
4. **Given** User A 收到主揪人留言通知，**When** 點擊該通知，**Then** 導航至 Event X 詳文頁、滾動至該則留言並套用高光特效

---

### Edge Cases

- 留言者自己不會收到自己觸發的通知
- 文章作者已透過既有 `post_new_comment` 通知涵蓋，不重複發送跟帖通知
- 使用者同時符合「活動參加者」與「活動留言者」兩種條件時，只收到一則通知（參加者身份優先，訊息為「你參加的活動…」）
- 活動主揪人同時也是留言者時，只收到一則「你主辦的活動…」通知（主揪人身份優先級最高）
- 主揪人不在活動的 participants 名單中（建立活動即代表會參加），「主揪人 + 參加者」的身份組合在系統中不存在
- 使用者取消參加活動後，不再收到「你參加的活動…」通知；但若曾留言，仍以「留言跟帖」身份收到「你留言過的活動…」通知（去重邏輯查詢當前狀態的自然結果）
- 通知對應的文章或活動已被刪除時，點擊通知仍導航至該頁面，由目標頁面處理不存在的情況
- 短時間內多則留言，每則獨立產生通知（不合併）
- 大量參加者的活動（如 50 人）一則留言產生大量通知時，仍應在合理時間內完成
- 新通知類型需整合既有的通知基礎設施（鈴鐺徽章即時更新、通知面板顯示、toast 即時提示、點擊標記已讀）

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 系統必須在文章收到新留言時，通知所有曾在該文章留過言的使用者（排除留言者本人與文章作者）
- **FR-002**: 系統必須在活動收到新留言時，通知該活動的所有參加者（排除留言者本人）
- **FR-003**: 系統必須在活動收到新留言時，通知所有曾在該活動留過言的使用者（排除留言者本人）
- **FR-004**: 系統必須在活動收到新留言時，通知該活動的主揪人（排除留言者本人即主揪人自己留言的情況）
- **FR-005**: 針對同一則留言事件，系統必須確保每位使用者最多只收到一則通知（跨 FR-001/002/003/004 去重）
- **FR-006**: 去重優先順序：文章作者的既有 `post_new_comment` > 活動主揪人通知 > 活動參加者通知 > 留言跟帖通知
- **FR-007**: 文章跟帖通知的訊息格式為「你留言過的文章『{文章標題}』有一則新的留言」
- **FR-008**: 活動主揪人通知的訊息格式為「你主辦的活動『{活動標題}』有一則新的留言」
- **FR-009**: 活動參加者留言通知的訊息格式為「你參加的活動『{活動標題}』有一則新的留言」
- **FR-010**: 活動留言跟帖通知的訊息格式為「你留言過的活動『{活動標題}』有一則新的留言」
- **FR-011**: 使用者不會收到自己觸發的通知
- **FR-012**: 新通知類型必須完整整合現有通知系統：鈴鐺徽章即時更新、通知面板顯示、toast 提示、點擊導航、標記已讀
- **FR-013**: 點擊文章跟帖通知必須導航至該文章頁面、滾動至該則留言並套用高光特效（沿用既有 `post_new_comment` 的 highlight 動畫）
- **FR-014**: 點擊活動留言通知（主揪人/參加者/跟帖）必須導航至該活動詳文頁、滾動至該則留言並套用高光特效（與文章通知相同的 highlight 動畫）

### Key Entities

- **通知 (Notification)**: 延伸現有通知實體，新增四種觸發場景
  - 文章跟帖留言：接收者為該文章的歷史留言者（排除文章作者與留言者本人）
  - 活動主揪人留言：接收者為該活動的建立者/主揪人（排除留言者本人即主揪人自己留言）
  - 活動參加者留言：接收者為該活動的參加者（排除留言者本人與主揪人，因主揪人已有專屬通知）
  - 活動跟帖留言：接收者為該活動的歷史留言者（排除留言者本人、主揪人與參加者，因主揪人與參加者各有更高優先級的通知）
- **留言跟帖者 (Thread Participant)**: 概念實體——曾在某篇文章或活動留過言的不重複使用者集合，用於判定通知接收資格

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 使用者在相關文章或活動有新留言後 30 秒內，能在鈴鐺徽章看到未讀數量更新
- **SC-002**: 100% 的通知正確送達符合資格的接收者，不遺漏也不重複
- **SC-003**: 留言者本人不會收到任何自己觸發的通知
- **SC-004**: 同時符合多重通知條件的使用者，每次留言事件最多只收到一則通知
- **SC-005**: 活動有 50 位參加者時，一則留言的通知產生時間不影響留言操作的回應速度

## Assumptions

- 「留言過」的身份透過查詢留言記錄中不重複的作者來判定
- 活動主揪人永遠收到留言通知（類似文章作者收到 `post_new_comment`），不需報名參加或留言。若主揪人同時也是留言者，主揪人身份優先，只收到一則「你主辦的活動…」通知
- 主揪人不在活動的 participants 名單中（建立活動即代表會參加），故「主揪人 + 參加者」的身份組合在系統中不存在
- 去重優先順序：文章作者的既有 `post_new_comment` > 活動主揪人通知 > 活動參加者通知 > 留言跟帖通知
- 所有留言通知（文章跟帖、活動主揪人/參加者/跟帖）點擊後皆導航至目標頁面、滾動至該則留言並套用高光特效，沿用現有 `post_new_comment` 的 highlight 動畫
- 每則新留言獨立產生通知，不合併（例如 5 分鐘內有 3 則新留言，產生 3 組通知）
- 通知不區分活動是否已結束，已過期的活動有新留言仍發送通知
- 新通知類型完全整合現有通知基礎設施（鈴鐺、面板、toast、標記已讀），不需新增 UI 元件
- 目前系統無刪除留言功能，暫不處理「刪除留言後是否仍視為跟帖者」的邊界情況

## Out of Scope

- 通知合併（將多則來自同一實體的通知合併為一則摘要通知）
- 留言回覆（@提及特定使用者）通知
- 取消追蹤 / 退訂某篇文章或活動的留言通知
- 主揪人通知偏好設定（例如關閉活動留言通知）
- 「全部標記已讀」功能
